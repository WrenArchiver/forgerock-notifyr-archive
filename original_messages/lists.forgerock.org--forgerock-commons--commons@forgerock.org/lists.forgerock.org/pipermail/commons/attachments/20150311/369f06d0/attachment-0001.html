                                            <h2 style="margin: 10px 0 0 0; font-size: 20px; font-weight: normal; line-height: 1.500; margin: 40px 0 0 0"><a name="Summary" style="color: #3b73af; text-decoration: none"></a>Summary</h2> 
                                            <p style="margin: 10px 0 0 0">A merged set of requirements for the ForgeRock API descriptor follows.</p> 
                                            <p style="margin: 10px 0 0 0">This summarizes requirements expressed in the lists for each API Descriptor consumer.</p> 
                                            <p style="margin: 10px 0 0 0">The current understanding is based on existing description mechanisms in ForgeRock software, on what popular descriptor frameworks and SaaS providers offer, and on our understanding prior to implementation of the tools that would provide and consume API descriptors.</p> 
                                            <h3 style="margin: 10px 0 0 0; font-size: 16px; line-height: 1.563; margin: 30px 0 0 0"><a name="MustHave" style="color: #3b73af; text-decoration: none"></a>Must Have</h3> 
                                            <p style="margin: 10px 0 0 0"><b>Endpoints</b></p> 
                                            <ul> 
                                                <li>
                                                    Endpoint discovery (starting from a well-known endpoint)
                                                </li> 
                                                <li>
                                                    Identification of support for CREST operations (e.g. whether the endpoint supports update, paged results with description of guarantees, _queryId values, _queryExpression).
                                                </li> 
                                                <li>
                                                    Supported request &amp; response headers
                                                </li> 
                                                <li>
                                                    Success &amp; error status codes
                                                </li> 
                                                <li>
                                                    Supported parameters (including additional parameters)
                                                </li> 
                                                <li>
                                                    Resource schema
                                                </li> 
                                                <li>
                                                    Reuse and identification of common API descriptor components (e.g. clients can determine that certain endpoints are the same)
                                                </li> 
                                            </ul> 
                                            <p style="margin: 10px 0 0 0"><b>Resource Schema</b></p> 
                                            <ul> 
                                                <li>
                                                    JSON Schema for each JSON request &amp; response entity
                                                </li> 
                                                <li>
                                                    JSON Schema extensions for indicating field ordering, human-readable field label, whether a field is hidden/private, whether a field is searchable, how to look up an object referenced by the field
                                                </li> 
                                                <li>
                                                    Support for recursive component relationships (e.g. OpenAM realms, OpenIG Routes with sub-routes)
                                                </li> 
                                                <li>
                                                    Descriptions of complex structured JSON objects
                                                </li> 
                                                <li>
                                                    Indication whether an endpoint or action is public or not (e.g. /openam/json/session/...)
                                                </li> 
                                                <li>
                                                    Support for a broad range of syntaxes, equivalent to LDAP's support, including base64 encoded binary data
                                                </li> 
                                                <li>
                                                    Multipart form content support with supported MIME types
                                                </li> 
                                                <li>
                                                    Support for specifying how properties are matched (case sensitivity, case awareness, regex, etc.)
                                                </li> 
                                                <li>
                                                    Inheritance relationships (e.g. a user and an admin user, an abstract token and an OAuth2 token)
                                                </li> 
                                                <li>
                                                    Composition relationships (e.g. a user owning zero or more devices)
                                                </li> 
                                                <li>
                                                    Aggregation relationships (e.g. a user having a manager and being a member of groups)
                                                </li> 
                                                <li>
                                                    Constraints on properties (read-only, write-only, write-on-create-only, unique, mandatory, optional)
                                                </li> 
                                                <li>
                                                    &quot;Operational&quot; metadata properties (modify timestamps, ETags)
                                                </li> 
                                                <li>
                                                    Type information in resource instances (e.g. schema URN, equivalent to objectClass in LDAP)
                                                </li> 
                                                <li>
                                                    Support for specifying how resources are named (whether IDs are provided by the client or the server)
                                                </li> 
                                                <li>
                                                    Descriptions for actions per resource type (e.g. user resource supports a &quot;change password&quot; action, administrator user also has &quot;reset password&quot; and &quot;lock user account&quot; actions)
                                                </li> 
                                                <li>
                                                    Administration related hints (e.g. component must be restarted for changes to take effect)
                                                </li> 
                                                <li>
                                                    Default behaviors (default values for properties, whether defaults are inherited, etc.)
                                                </li> 
                                                <li>
                                                    Constraints between properties and components (e.g. an LDAP listener with SSL or StartTLS enabled has a &quot;key manager&quot; property identifying a Key Manager that must be enabled)
                                                </li> 
                                                <li>
                                                    Synopsis &amp; long descriptions for all documented elements
                                                </li> 
                                            </ul> 
                                            <p style="margin: 10px 0 0 0"><b>Versioning</b></p> 
                                            <ul> 
                                                <li>
                                                    API-level versioning (single server can expose different versions of the same API endpoint)
                                                </li> 
                                            </ul> 
                                            <p style="margin: 10px 0 0 0"><b>Tooling</b></p> 
                                            <ul> 
                                                <li>
                                                    Generation of descriptors for all CREST endpoints at run time from configuration data
                                                </li> 
                                                <li>
                                                    Validation for field values, schema
                                                </li> 
                                                <li>
                                                    Generation of API reference documentation from descriptors
                                                </li> 
                                                <li>
                                                    Developer tools to minimize duplication of information
                                                </li> 
                                                <li>
                                                    Server-side developer tooling for monitoring and task support (with both CREST and LDAP access in OpenDJ)
                                                </li> 
                                            </ul> 
                                            <p style="margin: 10px 0 0 0">To the extent that any information is provided by convention rather than part of the descriptor, all conventions must be clearly documented for each descriptor version:</p> 
                                            <p style="margin: 10px 0 0 0"><b>Per endpoint</b></p> 
                                            <ul> 
                                                <li>
                                                    API version
                                                </li> 
                                                <li>
                                                    Supported operations (CRUDPAQ)
                                                </li> 
                                                <li>
                                                    Realm Support
                                                </li> 
                                                <li>
                                                    Concurrency Control Enforcement (MVCC)
                                                </li> 
                                            </ul> 
                                            <p style="margin: 10px 0 0 0"><b>Per operation</b></p> 
                                            <ul> 
                                                <li>
                                                    Indication of what is required/optional
                                                </li> 
                                                <li>
                                                    (Non-resource) actions (Entirely separate description of input/output)
                                                </li> 
                                                <li>
                                                    Request (headers (Content-Type, etc.), query parameters, body)
                                                </li> 
                                                <li>
                                                    Response (possible response status codes, headers (Content-Type, etc.), body)
                                                </li> 
                                            </ul> 
                                            <h3 style="margin: 10px 0 0 0; font-size: 16px; line-height: 1.563; margin: 30px 0 0 0"><a name="NicetoHave" style="color: #3b73af; text-decoration: none"></a>Nice to Have</h3> 
                                            <ul> 
                                                <li>
                                                    Reuse existing JSON Schema (from OpenAM policies, OpenIDM managed objects &amp; connector object types)
                                                </li> 
                                                <li>
                                                    Descriptors generated at compile time with default configuration (for building release docs)
                                                </li> 
                                                <li>
                                                    Conditional requirements (e.g. if encryption is RSA, Certificate name required; if HMAC, shared secret required)
                                                </li> 
                                                <li>
                                                    Extensions to API descriptors with metainformation: 
                                                    <ul> 
                                                        <li>
                                                            UI hints: property ordering, whether a field is advanced or experimental or hidden, JS validation scripts
                                                        </li> 
                                                        <li>
                                                            Doc: stability levels
                                                        </li> 
                                                        <li>
                                                            Security: required authorization (OAuth2 scopes, roles, etc.)
                                                        </li> 
                                                    </ul> 
                                                </li> 
                                                <li>
                                                    Support for localization
                                                </li> 
                                                <li>
                                                    UI/CLI for managing data exposed (e.g. over Rest2LDAP)
                                                </li> 
                                                <li>
                                                    Command-line tools for installing and configuring servers
                                                </li> 
                                                <li>
                                                    GUI for visual development of initial configuration
                                                </li> 
                                                <li>
                                                    Examples/placeholders (for documentation)
                                                </li> 
                                            </ul> 
                                            <h3 style="margin: 10px 0 0 0; font-size: 16px; line-height: 1.563; margin: 30px 0 0 0"><a name="OutofScope" style="color: #3b73af; text-decoration: none"></a>Out of Scope</h3> 
                                            <ul> 
                                                <li>
                                                    Descriptors for arbitrary (non-CREST &amp; 3rd party, standard, etc.) HTTP APIs <br class="atl-forced-newline" /> One suggestion for handling these involves plug in support for one or more popular API descriptor frameworks, where the implementor is responsible for providing a descriptor.
                                                </li> 
                                                <li>
                                                    API authentication requirements <br class="atl-forced-newline" /> These are property of the deployment, typically not handled by the resource layer.
                                                </li> 
                                            </ul> 
                                            <h2 style="margin: 10px 0 0 0; font-size: 20px; font-weight: normal; line-height: 1.500; margin: 40px 0 0 0"><a name="RequirementsbyConsumer" style="color: #3b73af; text-decoration: none"></a>Requirements by Consumer</h2> 
                                            <p style="margin: 10px 0 0 0">API descriptor consumers include both the products and also our UIs, test tools, and doc tools.</p> 
                                            <h3 style="margin: 10px 0 0 0; font-size: 16px; line-height: 1.563; margin: 30px 0 0 0"><a name="OpenAM" style="color: #3b73af; text-decoration: none"></a>OpenAM</h3> 
                                            <p style="margin: 10px 0 0 0">Also see <a href="https://docs.google.com/a/forgerock.com/document/d/1Rciam2SbmBpj63RZPz1F6zEv8K7fgJ7VIjnn9zcBpVs" class="external-link" rel="nofollow" style="color: #3b73af; text-decoration: none">OpenAM API Descriptor requirements</a></p> 
                                            <p style="margin: 10px 0 0 0">Must have:</p> 
                                            <ul> 
                                                <li>
                                                    All CREST endpoints describable
                                                </li> 
                                                <li>
                                                    /json/authenticate endpoint describable - note, this is not a CREST endpoint, so we would expect at the very least to have to mirror any API descriptor functionality built into CREST in AM to support this.
                                                </li> 
                                                <li>
                                                    Users and Groups endpoints are only describable once AM is configured - different object classes selected for LDAP entities lead to different descriptions for those entities. Although there’s a plan to replace AM’s IdRepo implementations with IDM’s view aggregation functionality (made common) - in that case the users and groups implementations might just be re-exposing IDM endpoints?
                                                </li> 
                                                <li>
                                                    Manual documentation generation from API descriptions with ability to reference deployed URLs
                                                </li> 
                                                <li>
                                                    Endpoint discovery - we should be able hit a form of the endpoint (including un-bound router path fragments) to get back a list of supported endpoints. e.g. hitting /json?_api, or something, to get back the full list of endpoints as above.
                                                </li> 
                                                <li>
                                                    Non-duplication - we shouldn’t have to duplicate content between javadoc and API description. Perhaps support for converting @throws NotFoundException to a declaration of when a 404 is returned in the API description?
                                                </li> 
                                                <li>
                                                    Some way to support realms.
                                                </li> 
                                            </ul> 
                                            <p style="margin: 10px 0 0 0">Nice to have:</p> 
                                            <ul> 
                                                <li>
                                                    Reuse of existing JSON Schema generation endpoints as API description for policies.
                                                </li> 
                                                <li>
                                                    As AM is already using JSON Schema, it would be nice to use this to describe all APIs, extending its use to include JSON Schema Validation, which we don’t currently use.
                                                </li> 
                                            </ul>

                                            <h4 style="margin: 10px 0 0 0; font-size: 14px; line-height: 1.500; margin: 20px 0 0 0"><a name="REST2LDAP" style="color: #3b73af; text-decoration: none"></a>REST2LDAP</h4> 
                                            <p style="margin: 10px 0 0 0">Must have:</p> 
                                            <ol> 
                                                <li>
                                                    generating API descriptors at run time from configuration data
                                                </li> 
                                                <li>
                                                    describing complex structured JSON objects
                                                </li> 
                                                <li>
                                                    describing a broad range of syntaxes, equivalent to LDAP's support, and including base 64 encoded binary data
                                                </li> 
                                                <li>
                                                    specifying how attributes will be matched (case-sensitive, etc)
                                                </li> 
                                                <li>
                                                    inheritance relationships: a user and an admin user, or an abstract token and an OAuth2 token
                                                </li> 
                                                <li>
                                                    composition relationships: a user &quot;owning&quot; zero or more devices (parent-child relationships in LDAP)
                                                </li> 
                                                <li>
                                                    aggregation relationships: a user has a manager and is a member of certain groups
                                                </li> 
                                                <li>
                                                    constraining certain attributes: read-only, write-only, unique, mandatory, optional
                                                </li> 
                                                <li>
                                                    &quot;operational&quot; attributes (aka meta data): modify timestamps, etags
                                                </li> 
                                                <li>
                                                    associating type information with resource instances: could be a schema URN (equivalent to objectClass in LDAP)
                                                </li> 
                                                <li>
                                                    specifying how resources are named. In particular, whether resource IDs are client or server provided
                                                </li> 
                                                <li>
                                                    API versioning: resource models evolve and so do their schema, and it is likely that a single DJ deployment will want to expose different versions of the same API endpoint
                                                </li> 
                                                <li>
                                                    reusing and identifying common API descriptor components. It should be possible for an application interacting with OpenAM, OpenIDM, and OpenDJ to determine that their &quot;user&quot; endpoints are actually the same. This could be achieved by referencing descriptors externally by URL, or identifying inlined descriptors with a unique identifier such as a URN
                                                </li> 
                                                <li>
                                                    describing actions on a per resource type basis: a user resource may support a &quot;change password&quot; action, whereas an administrator (also a user) may additionally provide the ability to reset passwords and lock accounts belonging to other users
                                                </li> 
                                                <li>
                                                    describe any constraints on the standard CREST operations. For example, whether update operations are supported (some containers may be fully readonly), whether query requests support paged results, server side sorting, and query IDs.
                                                </li> 
                                            </ol> 
                                            <p style="margin: 10px 0 0 0">Nice to have</p> 
                                            <ol> 
                                                <li>
                                                    the ability to include additional extended information in an API descriptor. This may include meta information pertaining to: 
                                                    <ul> 
                                                        <li>
                                                            UIs such as the order in which fields should be displayed, whether fields are advanced or not, whether fields are experimental or hidden, JS validation scripts
                                                        </li> 
                                                        <li>
                                                            documentation: stability levels
                                                        </li> 
                                                        <li>
                                                            security: required authorization including OAuth2 scopes, roles, etc. Unlike authentication, authorization is quite often the responsibility of the resource server
                                                        </li> 
                                                    </ul> 
                                                </li> 
                                                <li>
                                                    localization support: the API descriptor is likely to include documentation such as field descriptions. It should be possible to request localized versions of this.
                                                </li> 
                                                <li>
                                                    a UI/CLI for managing data exposed over Rest2LDAP
                                                </li> 
                                            </ol> 
                                            <p style="margin: 10px 0 0 0">Not required:</p> 
                                            <ol> 
                                                <li>
                                                    server side compile time development language support such as Java annotations: Rest2LDAP is data driven
                                                </li> 
                                                <li>
                                                    the ability to describe authentication requirements. Authentication requirements are a property of the deployment and typically handled above the resource layer and may depend on who is accessing the resources and how (e.g. WAN or LAN). Therefore I think they should be fully decoupled from API descriptors.
                                                </li> 
                                            </ol> 
                                            <h4 style="margin: 10px 0 0 0; font-size: 14px; line-height: 1.500; margin: 20px 0 0 0"><a name="AdministrationConfigurationModel" style="color: #3b73af; text-decoration: none"></a>Administration Configuration Model</h4>

                                            <ol> 
                                                <li>

                                                <li>

                                                </li> 
                                                <li>
                                                    configuration reference guide and manual pages, although these can be derived automatically from the CLI
                                                </li> 
                                                <li>
                                                    configuration UI - administrators generally configure OpenDJ using a CLI so I don't think this is a high priority requirement. However, we'd get it for free (or very cheaply!) if we have sufficient tooling for API descriptors
                                                </li> 
                                                <li>
                                                    server side developer tooling such as annotations support: we already have a powerful framework for the OpenDJ configuration, but a modern annotation/binding based approach would be even more developer friendly. No such framework exists for developing new monitoring and administrative tasks capabilities, so server side tooling would be of benefit
                                                </li> 
                                                <li>
                                                    maintain LDAP access to our configuration, monitoring, and tasks. It may be possible to switch our task support from LDAP to HTTP, but I think that doing something similar for monitoring and configuration could have a large impact on existing users (TBC with Ludo).
                                                </li> 
                                            </ol>

                                            <p style="margin: 10px 0 0 0">API descriptors must contain enough information that a tool can derive the following for HTTP-level API reference documentation:</p> 
                                            <ul> 
                                                <li>
                                                    API version
                                                </li> 
                                                <li>
                                                    API access (if descriptor also covers non-public elements)
                                                </li> 
                                                <li>
                                                    API description
                                                </li> 
                                                <li>
                                                    Per API endpoint 
                                                    <ul> 
                                                        <li>
                                                            Requests 
                                                            <ul> 
                                                                <li>
                                                                    Valid methods
                                                                </li> 
                                                                <li>
                                                                    Headers, query string parameter, fragment, entity (resource)
                                                                </li> 
                                                                <li>
                                                                    Constraints, defaults, optional examples
                                                                </li> 
                                                            </ul> 
                                                        </li> 
                                                        <li>
                                                            Responses (status codes, headers, entity)
                                                        </li> 
                                                        <li>
                                                            Entity (resource type, schema, relationship to other resources) 
                                                            <ul> 
                                                                <li>
                                                                    Format of the resource (JSON, XML, etc.)
                                                                </li> 
                                                            </ul> 
                                                        </li> 
                                                    </ul> 
                                                </li> 
                                                <li>
                                                    Synopsis &amp; full descriptions for all elements
                                                </li> 
                                                <li>
                                                    Nice to have: security scheme info for authn/authz
                                                </li> 
                                            </ul> 
                                            <p style="margin: 10px 0 0 0">Summary of requirements for the documentation tool that consumes API descriptors &amp; produces docs:</p> 
                                            <ul> 
                                                <li>
                                                    Can derive all necessary information from API descriptor
                                                </li> 
                                                <li>
                                                    Allows API discovery for each supported version
                                                </li> 
                                                <li>
                                                    Allows interaction with service
                                                </li> 
                                                <li>
                                                    Uses skin for customization
                                                </li> 
                                                <li>
                                                    Uses hypertext to describe resources
                                                </li> 
                                                <li>
                                                    Supports recursive path templates (/[{realm}/]*users/{uid})
                                                </li> 
                                                <li>
                                                    Supports polymorphism (user resource definition depends on realm)
                                                </li> 
                                                <li>
                                                    Supports interdependent parameters (other parameters depend on _queryId value)
                                                </li> 
                                                <li>
                                                    Supports i18n
                                                </li> 
                                            </ul>
