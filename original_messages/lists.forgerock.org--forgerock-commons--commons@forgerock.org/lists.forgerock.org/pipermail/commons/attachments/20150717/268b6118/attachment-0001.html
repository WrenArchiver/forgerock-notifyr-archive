                                        <td class="text-paragraph-pattern-container mobile-resize-text " style="padding: 0px; border-collapse: collapse; padding: 0 0 10px 0"> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">While testing an upgrade to support AES-256-CBC/HMAC-SHA-512 mode in JWE encryption, I added some test cases based on the examples in the spec RFC: <a href="https://tools.ietf.org/html/rfc7518#appendix-B" class="external-link" rel="nofollow" style="color: #3b73af; text-decoration: none">https://tools.ietf.org/html/rfc7518#appendix-B</a> . Unfortunately, the current code gives incorrect answers for the HMAC tags that are computed. This is because the data given in the RFC test cases is raw bytes and not valid UTF-8, but our signing approach attempts to convert the authentication data payload into a String using UTF-8 charset and then back again. This results in different bytes being passed to the HMAC and an incorrect MAC is then produced.</p> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">As the data payload includes non-text content (e.g., unsigned 64-bit length fields, random IV data, etc) it is not appropriate to attempt to convert this to a string in any encoding that might drop/alter byte sequences.</p> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">Suggested solution is to add a new sign() method on the SigningHandler that takes raw bytes rather than a String.</p>
