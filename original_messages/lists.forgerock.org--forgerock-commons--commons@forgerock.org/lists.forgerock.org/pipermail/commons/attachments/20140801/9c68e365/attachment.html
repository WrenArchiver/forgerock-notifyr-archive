    <td style="color:#000000;font-family:Arial,FreeSans,Helvetica,sans-serif;font-size:12px;padding:0 0 10px 0;width:100%;">
        <p style='margin-top:0;margin-bottom:10px;'>Currently the _queryId / _queryFilter / _queryExpression parameters are only "officially" recognized as part of a GET (resulting in a standard query request). A beneficial enhancement would be to allow these parameters to also be provided as part of a PATCH or a DELETE operation, which could allow the resource to perform actions on a record (or records) without having to issue individual requests for each record (based on a known ID value). This provides two main benefits:</p>

<p style='margin-top:0;margin-bottom:10px;'>1) Making certain single-record updates / deletes more efficient (as they could be performed in a single REST call instead of a call to first find the ID and then a separate call to PATCH or DELETE with that ID)</p>

<p style='margin-top:0;margin-bottom:10px;'>2) Enabling the service to update or delete multiple records in a single operation. For example, in a DELETE you may wish to delete a large number of records in a single operation, because doing so one-at-a-time would be hugely inefficient. This would be equivalent to issuing a SQL query like so:</p>
<div class="code panel" style="border-width: 1px;"><div class="codeContent panelContent">
<pre class="code-java">DELETE FROM table WHERE a &gt; 1</pre>
</div></div>
<p style='margin-top:0;margin-bottom:10px;'>Rather than having N+1 queries to the database, where N is the number of records returned from the one SELECT query (this is what we have to do now).</p>

<p style='margin-top:0;margin-bottom:10px;'>Similarly, it would be very useful to be able to update multiple records with a PATCH definition. For example, you may wish to mark a large number of records as inactive rather than deleting them. You could do this with a single PATCH-by-query request just as you would have done with a DELETE-by-query request, and for the same reasons outlined above this would be far more efficient. </p>

<p style='margin-top:0;margin-bottom:10px;'>Also, given that transactions can be problematic through REST calls, enabling a single command to execute would be transactional by default, and so the traditional problems of transaction support may be able to be mitigated.</p>
