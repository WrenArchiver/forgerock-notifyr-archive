        <div class="comment-block" style="background-color:#edf5ff;border:1px solid #dddddd;color:#000000;padding:12px;"><p>If I understand correctly there are a few parts to this request:</p>

<p>1. More information in the transformed JSON response. <br/>
2. Logging of exceptions</p>

<p>In regards to 1), please identify the information specifically that would seem desirable. Stack traces are deliberately by default not provided to clients, as they can be seen as a security risk. We could consider a server side flag to optionally include them, e.g. for development purposes</p>

<p>In regards to 2), many failures should be handled by the client (and decided if it needs logging), and should not fill up the server side logs. However, for example in OpenIDM we always log 5xx exceptions in the server side log. Maybe that could be considered as a cross-cutting filter that can be used (and configured) across the suite.</p>

<p>Such a (configurable) cross-cutting filter would also allow to optionally log all exceptions for development purposes. Note that as we introduce scripted filters across the product it also would become more trivial to quickly add an onError script that just logs for example.</p>

<p>In short, by default we should not server side log (with stack traces etc) every application level error. But there is a class of server side (unexpected) errors we can do so, and we should provide simple ways to add explicit logging of the errors if desired.</p>
