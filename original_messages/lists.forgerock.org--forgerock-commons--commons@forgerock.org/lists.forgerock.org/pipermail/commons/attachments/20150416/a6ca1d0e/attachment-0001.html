                                <table class="text-paragraph-pattern" cellspacing="0" cellpadding="0" border="0" width="100%" style="border-collapse: collapse; mso-table-lspace: 0pt; mso-table-rspace: 0pt; font-family: Arial, sans-serif; font-size: 14px; line-height: 20px; mso-line-height-rule: exactly; mso-text-raise: 2px">

                                        <td class="text-paragraph-pattern-container mobile-resize-text " style="padding: 0px; border-collapse: collapse; padding: 0 0 10px 0"> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">Dialogs may be defined in such a way that they have their own routes, so that when they are opened the browser's hash reflects that state. The expectation is that a user could refresh their browser at that point and expect to be returned to (more or less) the same state they were in - with the expected dialog open, at least.</p> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">Some dialogs may also be associated with a particular base view - for example, the forgerock-ui-mock project has a &quot;self-registration&quot; page which includes a link to a &quot;Terms of Use&quot; dialog. Opening this dialog always also implies that that &quot;self-registration&quot; page will be rendered beneath it.</p> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">There is presently a problem with the framework when a route for a dialog such as described above is loaded directly (rather than navigated to during the course of using the application). The problem is that some base views expect to be rendered with particular parameters to be provided to it; when rendered as the base view for a dialog, these parameters are not provided. This easily lead to a broken rendering of the base view.</p> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">To resolve this problem, I propose a convention be established for dialogs and base views which use parameters:</p> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">1) A dialog that refers to a base view with parameters MUST also accept those same parameters and in the same order.<br /> 2) A dialog that refers to a base view with parameters MAY define additional parameters to be provided to itself.<br /> 3) A base view with arguments MUST be capable of handling additional values in the args array provided to its render function; these should probably simply be ignored.</p> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">Given this convention, a route for a dialog using a parameterized based view should be able to be opened directly in the browser; the parameters parsed from the dialog route will be passed into the render function for the base view, and because (by the above convention) they align with each other, everything should render properly.</p>
