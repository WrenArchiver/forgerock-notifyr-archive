                                            <p style="margin: 10px 0 0 0"><span class="error">[Work in progress]</span></p> 
                                            <p style="margin: 10px 0 0 0">OpenIG will have two categories of REST APIs:</p> 
                                            <ol> 
                                                <li>
                                                    administration APIs: configuring, monitoring, tasks. We have control over these, so they should be CREST endpoints
                                                </li> 
                                                <li>
                                                    3rd party endpoints: these represent the external services that OpenIG is protecting. We have no control over them so they could be just about anything, even plain old static HTML. It may be that some protected resources are REST APIs that could be described in some way.
                                                </li> 
                                            </ol> 
                                            <p style="margin: 10px 0 0 0">The short term goal is to address the administration requirements (#1 above).</p> 
                                            <ul> 
                                                <li>
                                                    configuration 
                                                    <ul> 
                                                        <li>
                                                            config model is defined statically at compile time by the developer
                                                        </li> 
                                                        <li>
                                                            developer should be able to focus on business logic, not boilerplate config parsing
                                                        </li> 
                                                        <li>
                                                            component needs to be notified when its config changes or when related component's configuration changes. E.g. an HTTP client should be notified when its key manager is changed. The component effectively has a read-only view of the config. It's best if config is decoupled from component in order to support atomic changes. Individual config parameters should not be injected one at a time into a live component. Whilst the config framework should perform basic schema checking and referential integrity, it is not expected to do everything. For example, it might check whether a port number is within range 0-65535 but it might not check if the port is free
                                                        </li> 
                                                        <li>
                                                            the mechanism used by developers to describe the API must be as developer friendly as possible, otherwise it won't be used effectively.
                                                        </li> 
                                                        <li>
                                                            a MUST HAVE is a lightweight CLI for quickly deploying new instances. This is a requirement for elasticity in the cloud, increasing use of devops patterns like &quot;<a href="http://martinfowler.com/bliki/PhoenixServer.html" class="external-link" rel="nofollow" style="color: #3b73af; text-decoration: none">Phoonix Server</a>&quot;, increasing use of tools like Puppet and Docker, etc
                                                        </li> 
                                                    </ul> 
                                                </li> 
                                            </ul> 
                                        </td>
