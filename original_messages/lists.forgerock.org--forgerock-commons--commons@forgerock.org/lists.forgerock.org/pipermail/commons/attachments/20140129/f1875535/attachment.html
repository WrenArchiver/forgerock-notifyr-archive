        <div class="comment-block" style="background-color:#edf5ff;border:1px solid #dddddd;color:#000000;padding:12px;"><p>In fact, I think that using the $ character to identifier field values which are references to mime-parts is still risky because it will cause automatic inlining of $ref values (see <a href="https://tools.ietf.org/html/draft-pbryan-zyp-json-ref-03">https://tools.ietf.org/html/draft-pbryan-zyp-json-ref-03</a>). Such attempts are likely to fail in the case where the intent is that the $ref is passed through to the underlying resource provider implementation.</p>

<p>I think that we have two choices:</p>

<ul>
        <li>use the already discussed mechanism but with an alternative character such as "@" for referencing related form-data, e.g:
<div class="preformatted panel" style="border-width: 1px;"><div class="preformattedContent panelContent">
<pre>POST /users?_action=create  HTTP/1.1
Host: example.com
Accept: application/json
Content-Type: multipart/form-data; boundary=kwkIqb-fsdbtcNpB4dJ_Xqf1-3b0Hp_VF9D0vsgL
Content-Length: ...

--kwkIqb-fsdbtcNpB4dJ_Xqf1-3b0Hp_VF9D0vsgL
Content-Disposition: form-data; name="json"
Content-Type: application/json; charset=US-ASCII
Content-Transfer-Encoding: 8bit

{
  "uid"             : "alice",
  "email"           : "alice@example.com",
  "@profilePhoto"    : "profilePhoto.content",
  "@profilePhotoType": "profilePhoto.mimetype",
  "@profilePhotoName": "profilePhoto.filename"
}
--kwkIqb-fsdbtcNpB4dJ_Xqf1-3b0Hp_VF9D0vsgL
Content-Disposition: form-data; name="profilePhoto"; filename="HappyJSON.jpg"
Content-Type: image/jpeg
Content-Transfer-Encoding: binary

*** binary image data ***
</pre>
</div></div></li>
        <li>use the $ref notation as described in <a href="https://tools.ietf.org/html/draft-pbryan-zyp-json-ref-03">https://tools.ietf.org/html/draft-pbryan-zyp-json-ref-03</a> but only inline $ref values which reference related form-data using relative URIs. This still seems a bit risky due to the inherent ambiguity resulting from the fact that some $ref values need resolving immediately in the Servlet, whereas others are left unchanged and passed through to the resource provider: what if some $ref values happen to look like form-data references? Perhaps we could identify them specifically by using the CID URI scheme as defined in <a href="http://tools.ietf.org/html/rfc2392">http://tools.ietf.org/html/rfc2392</a> or <a href="http://tools.ietf.org/html/rfc2557?">http://tools.ietf.org/html/rfc2557?</a> Can MIME form-data include a Content-ID field?
<div class="preformatted panel" style="border-width: 1px;"><div class="preformattedContent panelContent">
<pre>POST /users?_action=create  HTTP/1.1
Host: example.com
Accept: application/json
Content-Type: multipart/form-data; boundary=kwkIqb-fsdbtcNpB4dJ_Xqf1-3b0Hp_VF9D0vsgL
Content-Length: ...

--kwkIqb-fsdbtcNpB4dJ_Xqf1-3b0Hp_VF9D0vsgL
Content-Disposition: form-data; name="json"
Content-Type: application/json; charset=US-ASCII
Content-Transfer-Encoding: 8bit

{
  "uid"             : "alice",
  "email"           : "alice@example.com",
  "profilePhoto"    : { "$ref", "cid:profilePhoto#content" },
  "profilePhotoType": { "$ref", "cid:profilePhoto#mimetype" },
  "profilePhotoName": { "$ref", "cid:profilePhoto#filename" },
}
--kwkIqb-fsdbtcNpB4dJ_Xqf1-3b0Hp_VF9D0vsgL
Content-Disposition: form-data; name="profilePhoto"; filename="HappyJSON.jpg"
Content-Type: image/jpeg
Content-Transfer-Encoding: binary

*** binary image data ***
</pre>
</div></div></li>
</ul>
