                                            <p style="margin: 10px 0 0 0">Hi <a href="https://bugster.forgerock.org/jira/secure/ViewProfile.jspa?name=alin" class="user-hover" rel="alin" style="color: #3b73af; text-decoration: none">alin</a> - thanks for raising this issue and the in depth analysis that you have performed. I think that there is definitely an issue here that exposes a number of problems.</p> 
                                            <p style="margin: 10px 0 0 0">Before migrating to CHF we used the Servlet API directly. Using Servlet the container would provide the webapp with an HttpServletResponse which includes the response's output stream. The container would let the webapp write to the output stream and then flush the output to the network once the internal buffer is full. If the network TCP buffer is full then the webapp's thread would become blocked. To summarize, in CREST 2.x we could stream responses directly to the client and we were automatically throttled by speed of the client/network. However, this model is quite ugly because the webapp doesn't know when the buffer will be flushed and therefore must set HTTP response fields like headers and HTTP status code before writing any response content since the HTTP status code / headers must be written out before any HTTP content.</p> 
                                            <p style="margin: 10px 0 0 0">CHF has a slightly different programming model: it is the responsibility of the CHF application handling a response to allocate the response as well as a promise for the response. Specifically, the CHF Handler &quot;handle&quot; method must return a Promise:</p> 
                                            <div class="code panel" style="border-width: 1px;; border: 1px solid #ccc; background: #f5f5f5; font-size: 12px; line-height: 1.333; font-family: monospace; border: 1px solid #ccc; -moz-border-radius: 3px 3px 3px 3px; border-radius: 3px 3px 3px 3px; margin: 9px 0">
                                                <div class="codeContent panelContent" style="padding: 9px 12px"> 
                                                    <pre class="code-java" style="margin: 10px 0 0 0; max-height: 30em; overflow: auto; white-space: pre-wrap; word-wrap: normal">
Promise&lt;Response, NeverThrowsException&gt; handle(Context context, Request request) {
    PromiseImpl&lt;Response, NeverThrowsException&gt; promise = PromiseImpl.create();

    <span class="code-comment" style="color: #808080">// XXX - What happens here?
</span>
    <span class="code-keyword" style="color: #000091">return</span> promise;
}
</pre> 
                                                </div>
                                            </div> 
                                            <p style="margin: 10px 0 0 0">Note that the returned promise is effectively a finite queue due to the HTTP response's content output stream. There will never be any consumers of this queue until after we've returned the promise. If the implementation of the handle method attempts to write to the output stream before returning the promise then it will deadlock because no one is ever going to consume the data that it wrote. We &quot;solve&quot; this in CHF by using a buffer which allows the producer (the implementation of the &quot;handle&quot; method) to write data without blocking and deadlocking itself. However, we can't let the buffer grow too big (OOME), nor can we block the producer (deadlock), so we throw the overflow exception that you are seeing when the buffer is full.</p> 
                                            <p style="margin: 10px 0 0 0">However, there is only a risk of a deadlock if the implementation of the &quot;handle&quot; method above is synchronous (blocking). This may be the case when the handler is performing some query against an internal DB and streaming the results back to the client in the same thread. If the implementation of the method is asynchronous (non-blocking) then we will return the promise and the response without waiting for the response's content to be completed. This situation occurs when there is an async IO layer, such as a CHF HTTP client, DJ's LDAP SDK or the Rest2LDAP library which all act as async non-blocking network clients. In this case a separate thread will read data from the network and attempt to append it (e.g. as JSON) to the response's content stream. If the consumer is to slow to consume data from the content stream then the thread will get blocked.</p> 
                                            <p style="margin: 10px 0 0 0">So, we have two scenarios:</p> 
                                            <ol> 
                                                <li>
                                                    the &quot;handle&quot; implementation is blocking: response content must be buffered to avoid deadlock, but the buffer must not be too big otherwise there is a risk of OOME, nor can it block the producer
                                                </li> 
                                                <li>
                                                    the &quot;handle&quot; implementation is non-blocking: response content will be produced by a separate thread, so there is no risk of deadlock and the buffer can be finite and block the producer.
                                                </li> 
                                            </ol> 
                                            <p style="margin: 10px 0 0 0">What can we do?</p> 
                                            <ol> 
                                                <li>
                                                    I don't think the response's output stream should &quot;overflow&quot; as this will cause unexpected problems in both cases above if the client (consumer) cannot read data from the buffer fast enough
                                                </li> 
                                                <li>
                                                    I think the response's output stream should block the producer, but then there is a risk of deadlock if the handle method is the producer
                                                </li> 
                                                <li>
                                                    handle methods must not be producers. Instead, we must ensure that they are decoupled from the consumer side. The only way I can imagine us reliably doing that is if the handle implementation is run in a separate thread. In other words, I think that the method &quot;org.forgerock.json.resource.http.RequestRunner#visitQueryRequest&quot; should be executed in a thread pool.
                                                </li> 
                                            </ol> 
                                            <p style="margin: 10px 0 0 0">For (3) we could simply insist that implementations of CREST query must be non-blocking, but we cannot enforce the contract and it is likely to be abused. </p> 
                                            <p style="margin: 10px 0 0 0">In the short term there are a couple of other options:</p> 
                                            <ol> 
                                                <li>
                                                    re-use the buffer factory exposed in org.forgerock.http.HttpApplication#getBufferFactory since it can be configured by the application itself, e.g. to use a bigger buffer
                                                </li> 
                                                <li>
                                                    recommend usage of paged results for large result sets.
                                                </li> 
                                            </ol>
