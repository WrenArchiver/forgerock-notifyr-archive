                                                    <td class="text-paragraph-pattern-container mobile-resize-text " style="padding: 0px; border-collapse: collapse; padding: 0 0 10px 0"> <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0; margin-top: 0">Currently CHF is difficult to mock due to a combination of final classes with complex state and a lack of convenient construction facilities. As a HTTP framework, it is likely that most clients will want to mock interactions with the framework as far as possible.</p> <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">For example, to verify that a Response object is closed (a requirement of the API to avoid resource leaks) something like the following code must be used:</p> 
                                                        <div class="code panel" style="border-width: 1px;; border: 1px solid #cccccc; background: #f5f5f5; font-size: 12px; line-height: 1.333; font-family: monospace; border: 1px solid #cccccc; -moz-border-radius: 3px 3px 3px 3px; border-radius: 3px 3px 3px 3px; margin: 9px 0">
                                                            <div class="codeContent panelContent" style="padding: 9px 12px"> 
                                                                <pre class="code-java" style="margin: 10px 0 0 0; margin-top: 0; max-height: 30em; overflow: auto; white-space: pre-wrap; word-wrap: normal">
Response response = <span class="code-keyword" style="color: #000091">new</span> Response();
BranchingInputStream mockInputStream = mock(BranchingInputStream.class);
response.setEntity(mockInputStream);
...
verify(mockInputStream).close();
</pre> 
                                                            </div>
                                                        </div> <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">This is because (without Powermock) the Response object cannot be directly mocked and exposes no methods to introspect whether it has been closed or not. We must therefore examine the implementation details of the class to discover that if we pass a BranchingInputStream to the setEntity(Object) method then it will be directly stored and will be closed when the response is closed. This is clearly quite fragile and relies on tests knowing far too much about the details of the Response object.</p> <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">As the Response object is entirely mutable with public setters for all internal state, one possibility would be to simply remove the final modifiers from this and other classes as it is not clear what that is achieving anyway.</p> </td>
