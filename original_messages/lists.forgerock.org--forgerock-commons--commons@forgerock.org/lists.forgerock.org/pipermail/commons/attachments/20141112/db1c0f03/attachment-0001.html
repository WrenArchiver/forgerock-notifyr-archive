                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">From the discussion on CR-5177, it was indicated by several participants that a standard mechanism in CREST for doing password changes or changes of sensitive profile attributes (i.e., any attribute that can be used as a back-channel to reset the password). This facility is different from the existing update or patch requests as it requires providing existing credentials as well as the updated resource representation. In AM and IDM this is currently handled via (different) custom HTTP headers, which is not ideal from a common protocol standpoint.</p> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">A number of approaches have been proposed:</p> 
                                            <ul class="alternate" type="square"> 
                                                <li>
                                                    send current password as a query parameter (security issue, so should be discounted immediately)
                                                </li> 
                                                <li>
                                                    standardise some &quot;current password&quot; header across CREST (including support for Unicode passwords as per the original CR)
                                                </li> 
                                                <li>
                                                    standardise a &quot;changePassword&quot; / &quot;changeProtectedAttributes&quot; action
                                                </li> 
                                                <li>
                                                    standardise a &quot;_meta&quot; / &quot;header&quot; section in CREST resource bodies to allow credentials to be passed in the POST body
                                                </li> 
                                                <li>
                                                    provide a mechanism by which CREST actions/updates etc may trigger forced reauthentication (by whatever mechanism is used to protect the endpoint). In AM this would involve a &quot;forceAuth&quot; re-authentication. Might require temporarily invalidating the user's session or adding a one-time CSRF token to the request to ensure that authentication is done even if user is already logged in.
                                                </li> 
                                            </ul>
