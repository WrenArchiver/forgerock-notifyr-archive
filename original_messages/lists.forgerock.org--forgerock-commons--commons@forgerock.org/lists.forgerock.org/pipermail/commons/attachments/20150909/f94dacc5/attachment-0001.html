                                        <td class="text-paragraph-pattern-container mobile-resize-text " style="padding: 0px; border-collapse: collapse; padding: 0 0 10px 0"> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">I've marked this as critical because it must be addressed before feature freeze due to the API change.</p> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">OpenDJ is an LDAP server. When we are processing LDAP requests we are very likely to need context support similar to that provided in CHF. In particular, ClientContext, AttributesContext, SecurityContext, and even SessionContext, and probably many more. In addition, it would be nice if we could re-use some of the routing logic provided by AbstractRouter since we are likely to perform routing using various types of predicate, e.g. base DN, authorization state, client address, etc.</p> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">If we re-use these classes from CHF then the core of OpenDJ will look like it depends on HTTP somehow, which may confuse developers and anyone exposed to Java class names, e.g. via stack traces.</p> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0"><b>Suggested fix</b></p> 
                                            <ul> 
                                                <li>
                                                    consider moving these protocol agnostic types to a separate Java package, e.g. org.forgerock.services or org.forgerock.roa
                                                </li> 
                                                <li>
                                                    consider relocating the new Java package to forgerock-util or into a separate module or project. I think moving it to forgerock-util is sufficient.
                                                </li> 
                                            </ul> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">The only priority for me is the first point (separate Java package), since this is what will be most visible to end-users and developers.</p> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">Note that we won't be able to make these changes in minor releases of CHF. It may also be difficult to perform a major version upgrade of CHF in a minor release of DJ, hence my motivation for addressing this before 1.0.0 feature freeze.</p>
