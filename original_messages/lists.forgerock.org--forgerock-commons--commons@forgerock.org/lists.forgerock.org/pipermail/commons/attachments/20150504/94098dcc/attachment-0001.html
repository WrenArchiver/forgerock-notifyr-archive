                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">Quoting <a href="https://bugster.forgerock.org/jira/secure/ViewProfile.jspa?name=matthew" class="user-hover" rel="matthew" style="color: #3b73af; text-decoration: none">Matthew Swift</a></p> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">Thanks again <a href="https://bugster.forgerock.org/jira/secure/ViewProfile.jspa?name=brmiller" class="user-hover" rel="brmiller" style="color: #3b73af; text-decoration: none">Brendan Miller</a> for raising the ugly issue of failed vs successful promises, processing exceptions, etc. </p> 
                                            <h3 style="margin: 10px 0 0 0; font-size: 16px; line-height: 1.563; margin: 30px 0 0 0"><a name="Summary" style="color: #3b73af; text-decoration: none"></a>Summary</h3> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">Myself, <a href="https://bugster.forgerock.org/jira/secure/ViewProfile.jspa?name=guillaume.sauthier" class="user-hover" rel="guillaume.sauthier" style="color: #3b73af; text-decoration: none">guillaume.sauthier</a>, and <a href="https://bugster.forgerock.org/jira/secure/ViewProfile.jspa?name=laurent.vaills" class="user-hover" rel="laurent.vaills" style="color: #3b73af; text-decoration: none">Laurent Vaills</a> had a big discussion the other day (actually I think it took over a whole day) and I think that we have come to a concensus on the CHF API design. I'll summarize it here while Guillaume is updating his patch... </p> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">Originally G&amp;L were worried about two things: </p> 
                                            <ul> 
                                                <li>
                                                    when a filter/handler returns a <tt>Promise&lt;Response, ResponseException&gt;</tt> is is easy to react to a <tt>2xx</tt> response but even easier to forget to react to a <tt>non-2xx</tt> response represented by a <tt>ResponseException</tt>. Often in filters you want the same handling regardless of the response status, so it's a <b>pain to have two lots of call-backs</b>. Also, G&amp;L found the contract surprising: they expected the promise to succeed if a response was received regardless of the status code. They only expected an exception if some internal component failed for some reason (processing error)
                                                </li> 
                                                <li>
                                                    when writing client applications it is nice to be able to distinguish between a client side problem (e.g. <tt>IOException</tt>) and a response returned from the server, even if the response was a <tt>500</tt>.
                                                </li> 
                                            </ul> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">The <tt>Router</tt> presents a problem though: should a routing failure yield a <tt>404</tt> &quot;response&quot; or should it yield a processing exception? In G&amp;L's view it should really throw a processing exception, but this is obviously wrong, because the exception will bubble up and eventually be converted into a &quot;something bad happened&quot; <tt>5xx</tt> error in the front-end. The router should clearly yield a <tt>404</tt>. The same rule applies to many other types of filter/handler: an access control handler should clearly yield a <tt>403</tt> response. A proxy client communicating with a remote backend service should yield an appropriate &quot;gateway&quot; <tt>50x</tt> error if communication fails. The more we thought about it the more we realized that all filters should deal with &quot;errors&quot; in place by returning an appropriate response. Moreover, they should never throw a processing exception because, chances are, the front-end won't be able to react to it in an appropriate way anyway, e.g. by returning the wrong status code, or inappropriate content (http vs json). </p> 
                                            <h3 style="margin: 10px 0 0 0; font-size: 16px; line-height: 1.563; margin: 30px 0 0 0"><a name="Conclusion%231" style="color: #3b73af; text-decoration: none"></a>Conclusion #1</h3> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">Filters and Handlers, etc, should deal with errors and convert them to appropriate responses. They <b>should not &quot;throw exceptions over the fence&quot;</b> and let someone else deal with them </p> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">So that leaves us with our Promises... Should we continue to treat <tt>2xx</tt> responses as &quot;success&quot; and <tt>non-2xx</tt> responses as &quot;failure&quot;? In other words, is the original <tt>Promise&lt;Response, ResponseException&gt;</tt> API ok? We've pretty convinced now that <b>splitting responses into two categories is risky</b>. The new <tt>Status</tt> support that Laurent has added makes the case against splitting them even stronger: it's now really easy to perform conditional logic based on status codes. What sort of exception can be returned in a CHF Promise? Based on conclusion #1 we don't want components to throw any processing checked exceptions, so in fact a <b>CHF should never fail</b></p> 
                                            <h3 style="margin: 10px 0 0 0; font-size: 16px; line-height: 1.563; margin: 30px 0 0 0"><a name="Conclusion%232" style="color: #3b73af; text-decoration: none"></a>Conclusion #2</h3> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">CHF returns <tt>Promise&lt;Response, NeverThrowsException&gt;</tt></p> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">Admittedly, there's something a bit weird about a <tt>Promise&lt;Response, NeverThrowsException&gt;</tt>! One step at a time: perhaps we can make it &quot;prettier&quot; in a subsequent change. We also need a robust way to automatically deal with run-time exceptions in our handlers. This should be an integral part of the Promise API but is not addressed today. </p> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">We're still going to keep <tt>ResponseException</tt> because it simplifies control-flow inside async call-backs. As a developer I still want to be able to use try-catch blocks, even if my catch block converts the <tt>ResponseException</tt> to a <tt>Promise&lt;Response,...&gt;</tt></p>
