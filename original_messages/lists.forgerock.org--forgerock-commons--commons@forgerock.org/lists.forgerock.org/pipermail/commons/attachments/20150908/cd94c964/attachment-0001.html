                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">As a developer I want a single location in which authn/authz information about the end-user is store. This information is very likely to be a requirement for both CREST and non-CREST HTTP endpoints, therefore it should be moved into CHF. In particular, it is very likely that a CHF filter (previously a Servlet filter) will be responsible for populating the SecurityContext. For example, an OpenID Connect Filter will populate the SecurityContext with authorization information. Resource server implementations, usually CREST based, will then use the authz information for performing access control decisions.</p> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">Suggested fix:</p> 
                                            <ul> 
                                                <li>
                                                    move SecurityContext from CREST to CHF
                                                </li> 
                                                <li>
                                                    move SecurityContextFactory to CHF's Servlet integration module so that we can integrate with existing non-CHF components such as AM policy agents.
                                                </li>
