                                            <p style="margin: 10px 0 0 0">Copy paster from <a href="http://sources.forgerock.org/cru/CMN-189#c86855" class="external-link" rel="nofollow" style="color: #3b73af; text-decoration: none">http://sources.forgerock.org/cru/CMN-189#c86855</a></p> 
                                            <p style="margin: 10px 0 0 0">Maybe I could give a little background on the OpenIG case:</p> 
                                            <p style="margin: 10px 0 0 0">The ultimate need is to be able to separate admin traffic from functional traffic.<br /> That means that we'll need to have an HTTP connector for serving functional traffic and another one for admin traffic (so that we have separated thread pools, different listening network interfaces and port).<br /> With this isolation, we can mitigate DOS attacks.</p> 
                                            <p style="margin: 10px 0 0 0">So, because we want traffic isolation, we shouldn't share the same <tt>HttpApplication</tt> for normal and admin traffic.<br /> So here we come with the multi-application requirement.</p> 
                                            <p style="margin: 10px 0 0 0">That said, there are still some links between the admin and normal HttpApplications:</p> 
                                            <ul> 
                                                <li>
                                                    the admin is configuring the other one (push new routes, configure components, monitor, ...)
                                                </li> 
                                                <li>
                                                    the functional app is probably sending events used for monitoring to the admin or some intermediary
                                                </li> 
                                            </ul>

                                            <p style="margin: 10px 0 0 0">That means that we still need to share something between theses instances.<br /> Here, a ContextListener (like what you wrote) would be a good match because you instantiate yourself the <tt>HttpApplication</tt> instances and you have the freedom to pass constructor parameters to share objects between theses elements:</p> 
                                            <div class="code panel" style="border-width: 1px;; border: 1px solid #ccc; background: #f5f5f5; font-size: 12px; line-height: 1.333; font-family: monospace; border: 1px solid #ccc; -moz-border-radius: 3px 3px 3px 3px; border-radius: 3px 3px 3px 3px; margin: 9px 0">

</span>HttpApplication admin = <span class="code-keyword" style="color: #000091">new</span> AdminHttpApplication(shared);
HttpApplication gateway = <span class="code-keyword" style="color: #000091">new</span> GatewayHttpApplication(shared);
</pre>

                                            <p style="margin: 10px 0 0 0">Right now, with CHF and its current Servlet integration, it's impossible to reach that goal AFAIK:<br /> If we want to listen onto different ports/interfaces we would have to have 2 different webapps, each of them with 1 http application, but then its becoming much harder (if not impossible) to share object instances between the admin and the gateway elements.</p> 
                                            <p style="margin: 10px 0 0 0">This requirement mandates to have more control than what we can have in a servlet environment with a standard war.</p> 
                                            <p style="margin: 10px 0 0 0">In order to prepare this future, we would like to be able to support multi-app right now (even if that does not give us the guarantees we're looking for against DOS for instance).<br /> That will make a future migration pain-less.</p>
