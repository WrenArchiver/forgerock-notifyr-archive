                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">I don't know why, but I just don't like the resource-core module. The name &quot;resource-core&quot; sounds like it is part of CREST that has got lost in CHF. Renaming the module to http-something doesn't seem reasonable either because it's not http specific either.</p> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">I understand the motivation for it though: to isolate code which is common to both CREST and CHF without introducing a dependency between CREST on CHF.</p> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">I see a number of options:</p> 
                                            <ol> 
                                                <li>
                                                    Matt: get over it <img class="emoticon" src="https://bugster.forgerock.org/jira/images/icons/emoticons/smile.gif" height="16" width="16" align="absmiddle" alt="" border="0" />
                                                </li> 
                                                <li>
                                                    merge it into http-core and assume that all CREST applications will ultimately use CHF anyway
                                                </li> 
                                                <li>
                                                    rename the module to something a bit more aligned with the rest of CHF, e.g. http-util. But this module is neither HTTP specific nor is it CREST specific, so the name &quot;resource-xxx&quot; is not really appropriate.
                                                </li> 
                                            </ol> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">My preference is for (2). The routing support should be part of the CHF core, meaning that classes like the Router, ResourceName (ResourcePath), API versioning, etc will be in http-core. In addition, I think all CREST applications will probably use CHF anyway, so the extra baggage is not such a big deal.</p>
