        <div class="comment-block" style="background-color:#edf5ff;border:1px solid #dddddd;color:#000000;padding:12px;"><p>Note that JASPI uses an HTTP header to communicate ATTRIBUTE_AUTH_PRINCIPAL whereas SecurityContextFactory uses request attributes for both pieces of information.</p>

<p>Myself and Phill have agreed that both components should use attributes to communicate this information. The original reason from Jonathan for using a header was  because of having the auth filter in front of a reverse proxy as only the headers would get sent on but he is happy to have in that case a custom context mapper filter which takes these request attributes and put them as headers in that scenario.</p>

<p>My, perhaps paranoid, concern with using a header is that there is a possible spoofing "attack" where a user, "alice", performs HTTP BASIC auth as "alice" but provides the ATTRIBUTE_AUTH_PRINCIPAL header "bob". Depending on the authentication mechanism in use (e.g. not JASPI), the header may not be cleared and may be misinterpreted by CREST which then logs the operation performed by Alice as having been done by Bob. Note that the authcid is generally informational and should not be used for authorization.</p>
