                                            <p style="margin: 10px 0 0 0"><a href="https://bugster.forgerock.org/jira/secure/ViewProfile.jspa?name=guillaume.sauthier" class="user-hover" rel="guillaume.sauthier" style="color: #3b73af; text-decoration: none">Guillaume Sauthier</a> - I agree.</p> 
                                            <p style="margin: 10px 0 0 0">I think that the precedence should be 1) parent, 2) parent contexts, 3) session/attribute properties. In other words, the name &quot;parent&quot; is reserved and overrides the session/attribute properties. The rationale for this is that scripts will be passed a generic Context object and will expect predictable behavior regardless of the type of the &quot;head&quot; Context:</p> 
                                            <div class="code panel" style="border-width: 1px;; border: 1px solid #ccc; background: #f5f5f5; font-size: 12px; line-height: 1.333; font-family: monospace; border: 1px solid #ccc; -moz-border-radius: 3px 3px 3px 3px; border-radius: 3px 3px 3px 3px; margin: 9px 0">
                                                <div class="codeContent panelContent" style="padding: 9px 12px"> 
                                                    <pre class="code-java" style="margin: 10px 0 0 0; max-height: 30em; overflow: auto; white-space: pre-wrap; word-wrap: normal">
context.parent <span class="code-comment" style="color: #808080">// what <span class="code-keyword" style="color: #000091; color: #808080">if</span> context is a session or attributes context?</span>
</pre> 
                                                </div>
                                            </div> 
                                            <p style="margin: 10px 0 0 0">The only question IMO is whether the priorities (2) and (3) are the right way round. I think they are because it works in the general case described above:</p> 
                                            <div class="code panel" style="border-width: 1px;; border: 1px solid #ccc; background: #f5f5f5; font-size: 12px; line-height: 1.333; font-family: monospace; border: 1px solid #ccc; -moz-border-radius: 3px 3px 3px 3px; border-radius: 3px 3px 3px 3px; margin: 9px 0">
                                                <div class="codeContent panelContent" style="padding: 9px 12px"> 
                                                    <pre class="code-java" style="margin: 10px 0 0 0; max-height: 30em; overflow: auto; white-space: pre-wrap; word-wrap: normal">
context.http <span class="code-comment" style="color: #808080">// what <span class="code-keyword" style="color: #000091; color: #808080">if</span> context is a session or attributes context? </span>
</pre> 
                                                </div>
                                            </div> 
                                            <p style="margin: 10px 0 0 0">References to &quot;context.http&quot; should predictably return the http context regardless of the type of &quot;context&quot;. However, it does mean that accessing attributes and session properties is risky since client code has no control over the names of parent contexts:</p> 
                                            <div class="code panel" style="border-width: 1px;; border: 1px solid #ccc; background: #f5f5f5; font-size: 12px; line-height: 1.333; font-family: monospace; border: 1px solid #ccc; -moz-border-radius: 3px 3px 3px 3px; border-radius: 3px 3px 3px 3px; margin: 9px 0">
                                                <div class="codeContent panelContent" style="padding: 9px 12px"> 
                                                    <pre class="code-java" style="margin: 10px 0 0 0; max-height: 30em; overflow: auto; white-space: pre-wrap; word-wrap: normal">
context.attributes.id <span class="code-comment" style="color: #808080">// what <span class="code-keyword" style="color: #000091; color: #808080">if</span> another component adds an <span class="code-quote" style="color: #009100; color: #808080">&quot;IdContext&quot;</span>?</span>
</pre> 
                                                </div>
                                            </div> 
                                            <p style="margin: 10px 0 0 0">IMO, flattening contexts for script-friendliness is always going to be risk prone for contexts like attributes/session whatever approach we take, so I think we should simply reduce the risk as much as possible. The only risk-free unambiguous approach is to split the session / attributes properties out in a sub-field:</p> 
                                            <div class="code panel" style="border-width: 1px;; border: 1px solid #ccc; background: #f5f5f5; font-size: 12px; line-height: 1.333; font-family: monospace; border: 1px solid #ccc; -moz-border-radius: 3px 3px 3px 3px; border-radius: 3px 3px 3px 3px; margin: 9px 0">

                                                    <pre class="code-java" style="margin: 10px 0 0 0; max-height: 30em; overflow: auto; white-space: pre-wrap; word-wrap: normal">
context.session.properties.id <span class="code-comment" style="color: #808080">// a bit verbose but ok I suppose.
</span>
context.attributes.properties.id <span class="code-comment" style="color: #808080">// verbose but looks like a tautology and is pretty ugly IMO</span>
</pre>
