        <p style='margin-top:0;margin-bottom:10px;'>See <a href="https://bugster.forgerock.org/jira/browse/OPENIDM-2260" title="Inconsistencies in encoding/decoding the IDs used of managed users">OPENIDM-2260</a>.</p>

<p style='margin-top:0;margin-bottom:10px;'>Suppose there is a resource with a + in the resource id: "b+jensen"</p>

<p style='margin-top:0;margin-bottom:10px;'>GET /users/b+jensen</p>

<p style='margin-top:0;margin-bottom:10px;'>will not work because the router will url-decode resource name paths as it routes, and <tt>ResourceName#urlDecode</tt> will decode the string if</p>
<div class="code panel" style="border-width: 1px;"><div class="codeContent panelContent">
<pre class="code-java">c == '+' || c == '%'</pre>
</div></div>
<p style='margin-top:0;margin-bottom:10px;'>is true for any character, c, in the string. (<tt>isUrlEscapeChar</tt>).</p>

<p style='margin-top:0;margin-bottom:10px;'>Likewise, <tt>ResourceName</tt> will encode characters which are not part of the <tt>SAFE_URL_CHARS</tt> BitSet, of which + is not one of them.</p>

<p style='margin-top:0;margin-bottom:10px;'>The bottom line is that + is a "safe" character in a URL path, does not need to be encoded, and certainly does not need to be decoded as a space.  Doing so</p>

<p style='margin-top:0;margin-bottom:10px;'>GET /users/b+jensen</p>

<p style='margin-top:0;margin-bottom:10px;'>into a read on the user "b jensen" which results in a 404.</p>

<p style='margin-top:0;margin-bottom:10px;'>Ref: <a href="http://stackoverflow.com/a/1006074">+ is only a special character in the query component</a></p>
