        <div class="comment-block" style="background-color:#edf5ff;border:1px solid #dddddd;color:#000000;padding:12px;"><p>The mockup is likely to be some drawings of the doc client, which might be single-page app that reads the API descriptor resource, and then presents a view of the resource to the reader.</p>

<p>This app should be generic to any CREST provider, with at least a ForgeRock skin. The app with a skin can then be used both in production to help developers work with concrete APIs served by actual deployments, and also in product documentation (for configuration API documentation, default CREST API references, and so forth).</p>

<p>For example, if a tenant gets access to the server at <a href="http://tenant.example.com/">http://tenant.example.com/</a>, then the service might expose an /apidocs endpoint to get the single-page app that reads the API descriptor resource exposed by the server (e.g. GET /?_action=describe&amp;profiles=examples). In another example, to publish a configuration reference for a product as part of the core docs, we generate the API descriptor resource at build time, and the doc is then the client packaged with that build-time API descriptor resource, published at <a href="http://docs.forgerock.org/product/version/config/">http://docs.forgerock.org/product/version/config/</a>.</p>

<p>Specifically for configuration tools reference documentation, we would instead want to generate doc source output (for example, DocBook XML) from command-line help. This would be separate, likely written into the CLI or GUI tools that rely on the CREST API descriptor resource for their metadata. That's because there are two levels of API involved: one API that's CREST directly, another that's mediated through a (higher-level) CLI or GUI tool.</p></div>
