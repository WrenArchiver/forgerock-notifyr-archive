        <div class="comment-block" style="background-color:#edf5ff;border:1px solid #dddddd;color:#000000;padding:12px;"><p>Actually, AFAIK it was never the intent that "AdviceContext" would become a generic means for communicating HTTP headers. I can see that is how it has been implemented though which is potentially dangerous in my opinion, see org.forgerock.json.resource.servlet.RequestRunner.writeAdvice():</p>

<div class="code panel" style="border-width: 1px;"><div class="codeContent panelContent">
<pre class="code-java"><span class="code-keyword">private</span> void writeAdvice() {
        <span class="code-keyword">if</span> (context.containsContext(AdviceContext.class)) {
            AdviceContext adviceContext = context.asContext(AdviceContext.class);
            <span class="code-keyword">for</span> (Map.Entry&lt;<span class="code-object">String</span>, <span class="code-object">String</span>&gt; entry : adviceContext.getAdvices().entrySet()) {
                httpResponse.setHeader(entry.getKey(), entry.getValue());
            }
        }
    }</pre>
</div></div>

<p>The reason this is bad is because it makes it easy for developers to subvert the CREST REST API contract and provide their own implementation defined behavior via the use of arbitrary headers. I'm not sure I understand why any CREST resource provider should be interested in cookies. Usually security information should intercepted externally by authentication filters and then propagated to the CREST servlet via a SecurityContext.</p>
