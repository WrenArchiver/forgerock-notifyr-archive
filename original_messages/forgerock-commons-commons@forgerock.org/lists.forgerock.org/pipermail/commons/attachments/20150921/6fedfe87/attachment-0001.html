                                        <td class="text-paragraph-pattern-container mobile-resize-text " style="padding: 0px; border-collapse: collapse; padding: 0 0 10px 0"> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">While working on <a href="https://bugster.forgerock.org/jira/browse/CREST-111" title="Implement CREST HTTP client" class="issue-link" data-issue-key="CREST-111" style="color: #3b73af; text-decoration: none">CREST-111</a> (HTTP Client) we encountered a design problem with the field filtering feature.</p> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">At the moment field filtering is performed in our front-ends, the InternalConnection and HttpAdapter<span class="error">[1]</span>, which means that a proxy component, such as the new HTTP client in <a href="https://bugster.forgerock.org/jira/browse/CREST-111" title="Implement CREST HTTP client" class="issue-link" data-issue-key="CREST-111" style="color: #3b73af; text-decoration: none">CREST-111</a>, is forced to request all fields in order to avoid double filtering<span class="error">[2]</span>. The other option is to clear out the request's fields parameter, e.g. request.getFields().clear() in order to prevent re-filtering in the front-end.</p> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">Front-ends should have a single responsibility and should not perform additional tasks such as field filtering. Filtering should only be performed for &quot;local&quot; data sources, those whose data is managed exclusively by the application, and should not be performed for remote data sources since the remote data source will perform filtering itself.</p> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0">Suggested fix:</p> 
                                            <ul> 
                                                <li>
                                                    deprecate the InternalConnectionFactory and related classes/methods
                                                </li> 
                                                <li>
                                                    implement a LocalRequestHandler, which factors out common processing steps for local data sources. The initial implementation will handle resource filtering but could be extended in future to provide plugin points for other common features, such as schema checking, fine grained access control, locking, caching, etc
                                                </li> 
                                                <li>
                                                    disable field filtering for the HttpAdapter and InternalConnectionFactory.
                                                </li> 
                                            </ul> 
                                            <p style="margin-top:0;margin-bottom:10px;; margin: 10px 0 0 0"><span class="error">[1]</span> the HttpAdapter is delegating the responsibility to InternalConnection<br /> <span class="error">[2]</span> causing excessive bandwidth utilization</p>
