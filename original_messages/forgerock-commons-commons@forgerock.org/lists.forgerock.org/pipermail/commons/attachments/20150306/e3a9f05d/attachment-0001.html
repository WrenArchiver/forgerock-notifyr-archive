                                            <h4 style="margin: 10px 0 0 0; font-size: 14px; line-height: 1.500; margin: 20px 0 0 0"><a name="ThefollowingcommentdocumentstherequirementsforOpenDJ%27sadministrationmodel." style="color: #3b73af; text-decoration: none"></a>The following comment documents the requirements for OpenDJ's administration model.</h4> 
                                            <p style="margin: 10px 0 0 0">At the moment OpenDJ does not expose any of its administration model over REST. Configuration, monitoring, and administrative tasks can all be performed remotely, but only over LDAP. </p> 
                                            <p style="margin: 10px 0 0 0">OpenDJ has a powerful and extensible configuration framework which currently includes approximately 180 different types of configurable component and over 500 individual properties. It would be impossible to manually maintain client tools and reference documentation for all of this. To that end the configuration framework includes an XML based DSL for describing components. These XML component descriptors are then consumed by our build tools which spit out server and client Java APIs, a reflection Java API, reference documentation. Our client tool, dsconfig, uses the generated APIs at run time to dynamically build its interface.</p> 
                                            <p style="margin: 10px 0 0 0">In terms of monitoring OpenDJ exposes a rich set of data via a single LDAP suffix whose entries correspond to individual components. The data is also exposed via JMX using the same underlying framework (LDAP and JMX are just front-end views of the monitoring framework). No tooling is provided for developing monitored components resulting in a system which is poorly specified, unstable, and difficult to document and test.</p> 
                                            <p style="margin: 10px 0 0 0">Finally, OpenDJ provides a &quot;task&quot; endpoint. Users add an LDAP entry representing a task beneath cn=tasks. OpenDJ decodes the tasks and runs the task internally notifying the task entry when the task's state has changed (e.g. when it completes). At the moment users interact with this endpoint using individual tools which are manually developed and maintained, e.g. the rebuild-index tool. Similar to the monitoring backend, there is no tooling provided for helping developers implement new tasks and their associated tools. Note that, unlike monitoring, users are not expected to interact with the tasks backend directly using LDAP: they are expected to use the tools.</p> 
                                            <p style="margin: 10px 0 0 0">It would be really nice if OpenDJ provided one or more CREST endpoints for administration. I think that we have the following high level requirements in order of priority:</p> 
                                            <ol> 
                                                <li>
                                                    integrate live monitoring data in web applications: this will always require reference documentation. In practice, I think that individual monitoring attributes will be selected for display by developers rather than mechanically read and displayed (i.e. administrators will only be interested in what they consider to be important statistics, not everything). Our monitoring data will be more &quot;consumable&quot; if it is exposed via a REST API
                                                </li> 
                                                <li>
                                                    a lightweight CLI for quickly deploying new instances. This is a requirement for elasticity in the cloud, increasing use of devops patterns like &quot;<a href="http://martinfowler.com/bliki/PhoenixServer.html" class="external-link" rel="nofollow" style="color: #3b73af; text-decoration: none">Phoenix Server</a>&quot;, increasing use of tools like Puppet and Docker, etc. We already have this today with dsconfig over LDAP
                                                </li> 
                                                <li>
                                                    configuration reference guide and manual pages, although these can be derived automatically from the CLI
                                                </li> 
                                                <li>
                                                    configuration UI - administrators generally configure OpenDJ using a CLI so I don't think this is a high priority requirement. However, we'd get it for free (or very cheaply!) if we have sufficient tooling for API descriptors
                                                </li> 
                                                <li>
                                                    server side developer tooling such as annotations support: we already have a powerful framework for the OpenDJ configuration, but a modern annotation/binding based approach would be even more developer friendly. No such framework exists for developing new monitoring and administrative tasks capabilities, so server side tooling would be of benefit
                                                </li> 
                                                <li>
                                                    maintain LDAP access to our configuration, monitoring, and tasks. It may be possible to switch our task support from LDAP to HTTP, but I think that doing something similar for monitoring and configuration could have a large impact on existing users (TBC with Ludo).
                                                </li> 
                                            </ol> 
                                            <p style="margin: 10px 0 0 0">Now in more detail.</p> 
                                            <h4 style="margin: 10px 0 0 0; font-size: 14px; line-height: 1.500; margin: 20px 0 0 0"><a name="Musthavesupportfor%3A" style="color: #3b73af; text-decoration: none"></a>Must have support for:</h4> 
                                            <ol> 
                                                <li>
                                                    all the must-have requirements for Rest2LDAP
                                                </li> 
                                                <li>
                                                    if we want to expose our configuration over REST then the API descriptor must be provide a similar level of functionality defined in our XML descriptor <a href="http://sources.forgerock.org/browse/opendj/trunk/opendj/opendj-maven-plugin/src/main/resources/config/stylesheets/admin.xsd?hb=true" class="external-link" rel="nofollow" style="color: #3b73af; text-decoration: none">schema</a>, which is currently used for describing components like our <a href="http://sources.forgerock.org/browse/opendj/trunk/opendj/opendj-maven-plugin/src/main/resources/config/xml/org/forgerock/opendj/server/config/LDAPConnectionHandlerConfiguration.xml?hb=true" class="external-link" rel="nofollow" style="color: #3b73af; text-decoration: none">LDAP connection handler</a> and our <a href="http://sources.forgerock.org/browse/opendj/trunk/opendj/opendj-maven-plugin/src/main/resources/config/xml/org/forgerock/opendj/server/config/LocalDBBackendConfiguration.xml?r=11737" class="external-link" rel="nofollow" style="color: #3b73af; text-decoration: none">JE backend</a>. In addition to the Rest2LDAP requirements we will need to express: 
                                                    <ul> 
                                                        <li>
                                                            administration related hints, such as whether a component will need the server to be restarted for changes to take effect
                                                        </li> 
                                                        <li>
                                                            default behaviors, including whether a property has a default value, whether it inherits default values from somewhere else, etc
                                                        </li> 
                                                        <li>
                                                            constraints between properties and components: for example, if the LDAP listener has SSL enabled or StartTLS enabled, then the &quot;key manager&quot; property must reference a key manager which is also enabled. OpenDJ's mechanism using an XML based condition language is possibly a bit over-engineered. It may be more flexible if conditions are included in API descriptors
                                                        </li> 
                                                    </ul> 
                                                </li> 
                                                <li>
                                                    server side developer tooling for monitoring and task support: an annotation based binding layer would seem like a good fit. This is must have because it provides a single point of reference between the server-side developers and all client side development.
                                                </li> 
                                            </ol> 
                                            <h4 style="margin: 10px 0 0 0; font-size: 14px; line-height: 1.500; margin: 20px 0 0 0"><a name="Nicetohave%3A" style="color: #3b73af; text-decoration: none"></a>Nice to have:</h4> 
                                            <ol> 
                                                <li>
                                                    all nice to have requirements for Rest2LDAP
                                                </li> 
                                                <li>
                                                    server side developer tooling for configuration: not so important for OpenDJ because it already has a configuration framework. Obviously a single common framework for all types of endpoints is desirable. If we don't migrate our config framework then we will need dynamic runtime generation of API descriptors
                                                </li> 
                                            </ol>
