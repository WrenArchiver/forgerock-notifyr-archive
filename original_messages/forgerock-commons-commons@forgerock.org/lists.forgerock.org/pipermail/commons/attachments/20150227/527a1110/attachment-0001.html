                                        <td class="text-paragraph-pattern-container mobile-resize-text " style="padding: 0px; border-collapse: collapse; padding: 0 0 10px 0"> 
                                            <h3 style="margin: 10px 0 0 0; font-size: 16px; line-height: 1.563; margin: 30px 0 0 0"><a name="DRAFTINPROGRESS" style="color: #3b73af; text-decoration: none"></a>DRAFT - IN PROGRESS</h3> 
                                            <p style="margin: 10px 0 0 0">tl;dr TODO</p> 
                                            <h3 style="margin: 10px 0 0 0; font-size: 16px; line-height: 1.563; margin: 30px 0 0 0"><a name="DocsusingAPIdescriptors" style="color: #3b73af; text-decoration: none"></a>Docs using API descriptors</h3> 
                                            <p style="margin: 10px 0 0 0">At least two types of documentation could rely on API descriptors.</p> 
                                            <ol> 
                                                <li>
                                                    API reference documentation
                                                </li> 
                                                <li>
                                                    Configuration reference documentation
                                                </li> 
                                            </ol> 
                                            <h4 style="margin: 10px 0 0 0; font-size: 14px; line-height: 1.500; margin: 20px 0 0 0"><a name="APIreferencedocumentation" style="color: #3b73af; text-decoration: none"></a>API reference documentation</h4> 
                                            <p style="margin: 10px 0 0 0">Developers are the audience for API reference documentation.</p> 
                                            <p style="margin: 10px 0 0 0">Developers need documentation that answers three types of questions:</p> 
                                            <ol> 
                                                <li>
                                                    <b>What is the API?</b> <br class="atl-forced-newline" /> API descriptors must hold the definitive answer to this type of question, which should be covered in the reference documentation.
                                                </li> 
                                                <li>
                                                    <b>How do I do such-and-such with this API?</b> <br class="atl-forced-newline" /> API descriptors should hold the answer to this type of question <em>when the answer is fully contained within the API</em>. (For example, How do I update only the manager property of a user profile?)
                                                </li> 
                                                <li>
                                                    <b>I encountered this error message. What should I do?</b> <br class="atl-forced-newline" /> API descriptors should make it possible to enumerate response status codes, but they do not answer this type of question. Ultimately, answering this type of question is a job for search engines, forums, StackOverflow, etc.
                                                </li> 
                                            </ol> 
                                            <p style="margin: 10px 0 0 0">Most developers access the APIs over HTTP, often using a language other than Java. The lowest-common-denominator representation of the API is HTTP requests and responses in this case. Indeed this is the way the sample CREST resource provider's API is described at <a href="http://commons.forgerock.org/forgerock-rest/json-resource-servlet/" class="external-link" rel="nofollow" style="color: #3b73af; text-decoration: none">http://commons.forgerock.org/forgerock-rest/json-resource-servlet/</a>.</p> 
                                            <p style="margin: 10px 0 0 0">We might have alternative (non-HTTP) transport for remote REST calls. (XMPP, CoAP, SNMP for batches?)</p> 
                                            <p style="margin: 10px 0 0 0">We might have SDK generators for language-specific bindings for an API. Such tools should conserve the reference information from the API descriptor in the layout expected for reference documentation in the target language.</p> 
                                            <p style="margin: 10px 0 0 0">Resources might not be JSON.</p> 
                                            <h4 style="margin: 10px 0 0 0; font-size: 14px; line-height: 1.500; margin: 20px 0 0 0"><a name="Configurationreferencedocumentation" style="color: #3b73af; text-decoration: none"></a>Configuration reference documentation</h4> 
                                            <p style="margin: 10px 0 0 0">All users read configuration reference documentation.</p> 
                                            <p style="margin: 10px 0 0 0">Unless they are developing a configuration tool, however, readers do not use the configuration API directly. Instead they use a GUI or CLI. UI requirements are a subject for <a href="https://bugster.forgerock.org/jira/browse/CREST-216" title="Gather API descriptor requirements for UIs" class="issue-link" data-issue-key="CREST-216" style="color: #3b73af; text-decoration: none">CREST-216</a>.</p> 
                                            <p style="margin: 10px 0 0 0">Briefly:</p> 
                                            <ul> 
                                                <li>
                                                    CLI configuration reference documentation, both usage and what would be man pages on UNIX, should fully describe all the arguments that you can provide to a command. <br class="atl-forced-newline" /> Reference manual pages can also include supplementary material (extra descriptions, explanations about dependencies, examples, see also links to other documentation). It is not clear that it makes sense to attach the supplementary information to the API descriptor.
                                                </li> 
                                                <li>
                                                    GUIs might call for reference documentation concerning the values you can set in form input fields. <br class="atl-forced-newline" /> Documentation for hand-written GUIs seems separate and outside the scope of the API descriptor.
                                                </li> 
                                            </ul> 
                                            <h3 style="margin: 10px 0 0 0; font-size: 16px; line-height: 1.563; margin: 30px 0 0 0"><a name="DocrequirementsAPIdescriptors" style="color: #3b73af; text-decoration: none"></a>Doc requirements API descriptors</h3> 
                                            <p style="margin: 10px 0 0 0">What is needed depends on the type of documentation.</p> 
                                            <h4 style="margin: 10px 0 0 0; font-size: 14px; line-height: 1.500; margin: 20px 0 0 0"><a name="RequirementsforAPIreferencedocumentation" style="color: #3b73af; text-decoration: none"></a>Requirements for API reference documentation</h4> 
                                            <p style="margin: 10px 0 0 0">API descriptors should be easy to find (well-known location in the deployed product).</p> 
                                            <p style="margin: 10px 0 0 0">API descriptors should reflect the concrete API at runtime.<br /> It would be nice to have the default API descriptors available without installing and configuring the service as well to build static docs for a product release.</p> 
                                            <p style="margin: 10px 0 0 0">API descriptors should make it possible to derive the following for API documentation:</p> 
                                            <ul> 
                                                <li>
                                                    API type (CREST, HTTP, ...)
                                                </li> 
                                                <li>
                                                    API version information
                                                </li> 
                                                <li>
                                                    API description, potentially including markup
                                                </li> 
                                                <li>
                                                    Enumeration of all parameters of valid HTTP requests 
                                                    <ul> 
                                                        <li>
                                                            E.g. when HTTP is the transport: paths, query string parameters, fragments, headers, entity (resource)
                                                        </li> 
                                                        <li>
                                                            Constraints on values
                                                        </li> 
                                                        <li>
                                                            Default values
                                                        </li> 
                                                        <li>
                                                            Example values
                                                        </li> 
                                                    </ul> 
                                                </li> 
                                                <li>
                                                    Enumeration of all responses (both success and errors) 
                                                    <ul> 
                                                        <li>
                                                            E.g. when HTTP is the transport: Status codes, headers, entity (resource)
                                                        </li> 
                                                    </ul> 
                                                </li> 
                                                <li>
                                                    For each resource 
                                                    <ul> 
                                                        <li>
                                                            Format of the resource (JSON, XML, etc.)
                                                        </li> 
                                                        <li>
                                                            Schema of the resource
                                                        </li> 
                                                        <li>
                                                            Relationships to other resources
                                                        </li> 
                                                        <li>
                                                            What operations can be performed on the resource
                                                        </li> 
                                                    </ul> 
                                                </li> 
                                                <li>
                                                    Descriptions for all elements, potentially including markup
                                                </li> 
                                            </ul> 
                                            <p style="margin: 10px 0 0 0"><b>Notes</b></p> 
                                            <ul> 
                                                <li>
                                                    On the implementation side, use of conventions can prevent the need to maintain boilerplate.
                                                </li> 
                                                <li>
                                                    If we define an HTTP type, could we write the descriptor by hand for non-CREST APIs?
                                                </li> 
                                                <li>
                                                    How would one describe an API like OpenAM's /json/authenticate?
                                                </li> 
                                                <li>
                                                    Maybe the API descriptor is not the right place to put transport-layer-specific information. And yet it should be possible to derive transport specific information from what is provided in the API descriptor. E.g. ConflictException maps to HTTP 409 Conflict.
                                                </li> 
                                                <li>
                                                    Authentication context, in HTTP headers that hold credentials, sessions, session handles, and status codes, seem like a separate problem distinct from describing an API. What about attaching something like <a href="http://raml.org/spec.html#security" class="external-link" rel="nofollow" style="color: #3b73af; text-decoration: none">RAML security schemes</a>, though, based on the implementation?
                                                </li> 
                                                <li>
                                                    Regarding descriptions, it feels like we need a short description (no markup) and optionally a full description (with markup). <br class="atl-forced-newline" /> The markup must be easy to read and write for simple cases, easy to read for more complex cases, with well-supported output for HTML and DocBook (for use in reference doc). Asciidoc might be a good fit.
                                                </li> 
                                                <li>
                                                    Also regarding descriptions, internationalization support would be good, though not if it forces the original description out into a resources bundle. If the doc is moved out of the code, many implementors will assume that maintaining the doc is someone else's problem.
                                                </li> 
                                            </ul> 
                                            <h4 style="margin: 10px 0 0 0; font-size: 14px; line-height: 1.500; margin: 20px 0 0 0"><a name="Requirementsforconfigurationreferencedocumentation" style="color: #3b73af; text-decoration: none"></a>Requirements for configuration reference documentation</h4> 
                                            <p style="margin: 10px 0 0 0">If the UI is built based on the configuration API descriptor, no doubt all of the constraints and defaults for values being set are handled by the UI.</p> 
                                            <p style="margin: 10px 0 0 0">It looks like the only requirement for configuration reference documentation is a description on any element. Here again, description could mean a short description (no markup, suitable for usage) and a full description (including markup, suitable for longer reference documentation).</p> 
                                            <h3 style="margin: 10px 0 0 0; font-size: 16px; line-height: 1.563; margin: 30px 0 0 0"><a name="APIDescriptorDocToolsRequirements" style="color: #3b73af; text-decoration: none"></a>API Descriptor Doc Tools Requirements</h3> 
                                            <p style="margin: 10px 0 0 0">Doc tools in this case are those that transform API descriptor content to user-friendly documentation.</p> 
                                            <p style="margin: 10px 0 0 0">Doc tools provide:</p> 
                                            <ul> 
                                                <li>
                                                    Runtime API and configuration reference documentation (online, CLI usage, etc.)
                                                </li> 
                                                <li>
                                                    Static API and configuration reference documentation
                                                </li> 
                                            </ul> 
                                            <p style="margin: 10px 0 0 0">On one hand, configuration reference documentation is less directly related to API descriptors than to the tools themselves.</p> 
                                            <ul> 
                                                <li>
                                                    Resource-level and field-level help in configuration GUIs should be part of the GUI.
                                                </li> 
                                                <li>
                                                    Configuration reference for a CLI could be just a wrapper for or an implementation of usage, no doubt with more information exposed.
                                                </li> 
                                            </ul> 
                                            <p style="margin: 10px 0 0 0">On the other hand, API reference documentation depends directly on the API descriptor.</p> 
                                            <p style="margin: 10px 0 0 0">Tools transforming API descriptor information to API reference documentation provide these features:</p> 
                                            <ul> 
                                                <li>
                                                    API discovery for developers (path hierarchy view, resource view)
                                                </li> 
                                                <li>
                                                    Interactive runtime documentation — read about it and try it out
                                                </li> 
                                                <li>
                                                    Customizable skin for runtime documentation
                                                </li> 
                                                <li>
                                                    Generated DocBook (Asciidoc?) reference source
                                                </li> 
                                                <li>
                                                    API version aware
                                                </li> 
                                                <li>
                                                    Awareness of API descriptor conventions
                                                </li> 
                                                <li>
                                                    Maps to supported API types (CREST, HTTP, ...) 
                                                    <ul> 
                                                        <li>
                                                            Entries for CREST verbs, not just HTTP methods (Create is PUT + id || POST + _action, Create and Update are both PUT)
                                                        </li> 
                                                        <li>
                                                            Sorting by CRUDPAQ, not just by HTTP method names
                                                        </li> 
                                                    </ul> 
                                                </li> 
                                                <li>
                                                    Hypertext resource descriptions
                                                </li> 
                                                <li>
                                                    Support for recursive path templates (<tt>/{realm}*/users/{uid</tt>})
                                                </li> 
                                                <li>
                                                    Support for polymorphism (user resource definition depends on realm)
                                                </li> 
                                                <li>
                                                    Support for interdependent parameters (other parameters depend on _queryId value)
                                                </li> 
                                                <li>
                                                    Support for i18n
                                                </li> 
                                                <li>
                                                    Support for hints about implementation-dependent authentication (e.g. RAML security schemes)
                                                </li> 
                                            </ul>
