                                            <p style="margin: 10px 0 0 0">The show and tell should focus on two areas of OpenDJ:</p> 
                                            <ul> 
                                                <li>
                                                    extensive <a href="https://wikis.forgerock.org/confluence/display/OPENDJ/Configuration+framework+overview" class="external-link" rel="nofollow" style="color: #3b73af; text-decoration: none">configuration framework</a> which uses XML component descriptors to generate client/server APIs, dynamically generated CLI (dsconfig), LDAP and JMX integration, localization support, generated config reference guide, manual pages, online help (part of the CLI). A lot of work went into designing the <a href="http://sources.forgerock.org/browse/opendj/trunk/opendj/opendj-maven-plugin/src/main/resources/config/stylesheets/admin.xsd?hb=true" class="external-link" rel="nofollow" style="color: #3b73af; text-decoration: none">XML descriptor schema</a> for modelling all aspects of OpenDJ's configuration, which now includes almost 180 types of configurable component. It has support for polymorphism, inheritance, composition, aggregation and, with the exception of dsconfig, support for recursive data models (e.g. OpenAM's realm/sub-realm model). It's focused on configuration models and predates JSON and Java annotations. If we were to implement this again today we'd avoid using XML and XSLT processing and, instead, let the server component developer define a configuration Bean with appropriate annotations, and derive everything from that.
                                                </li> 
                                            </ul> 
                                            <ul> 
                                                <li>
                                                    Rest2LDAP uses a &quot;mapping&quot; <a href="http://sources.forgerock.org/browse/opendj/trunk/opendj/opendj-rest2ldap-servlet/src/main/webapp/opendj-rest2ldap-servlet.json?r=9840#to168" class="external-link" rel="nofollow" style="color: #3b73af; text-decoration: none">configuration file</a> to specify the relationship between JSON resources and their associated LDAP entries. It's goal is to really define the transformation between the two protocols, not to describe the generated REST endpoint itself. However, it contrasts nicely with the static compile time nature of the config framework. Rest2LDAP is, instead, data driven in that it creates endpoints dynamically at run time from a configuration file. The configuration file should be sufficient to derive an API descriptor. The question is: should Rest2LDAP use as its configuration file an API descriptor extended (or &quot;profiled&quot;) to include information about the various resources/properties relate to LDAP?
                                                </li> 
                                            </ul> 
                                            <p style="margin: 10px 0 0 0">There are also several other aspects of OpenDJ which could benefit from this work:</p> 
                                            <ul> 
                                                <li>
                                                    it does not have a REST endpoint for configuring the server (only LDAP/JMX)
                                                </li> 
                                                <li>
                                                    it has monitoring and task LDAP endpoints which are not hooked into the config framework and, as a result, are hard to develop, maintain, and document. It would be really nice if the config, task (think &quot;actions&quot;), and monitoring LDAP endpoints were somehow unified. Whilst looking at a backend configuration I would like to easily be able to view (monitor) its status or rebuild an index
                                                </li> 
                                                <li>
                                                    it does not have a configuration UI, despite having an extensive CLI.
                                                </li> 
                                            </ul>
