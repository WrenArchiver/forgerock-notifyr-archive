                                            <p style="margin: 10px 0 0 0">In other words, you suggest that query string / form parameters would no more be part of the URI ?<br /> Note that in CREST 2.5, the request parameters/query string were never decoded (AFAICT). That's why you never had an exception I suppose.<br /> But that also means that you had to deal with encoded values (<tt>%20</tt>, ...) in the business logic. Any memories related to that ?</p> 
                                            <p style="margin: 10px 0 0 0">If input parameters are not compliant, there is nothing I can really do with that :'(</p> 
                                            <p style="margin: 10px 0 0 0">I see multiple options:</p> 
                                            <ul> 
                                                <li>
                                                    add a servlet filter in front of CHF Servlet that take care of fixing invalid query string values
                                                </li> 
                                                <li>
                                                    do the same in CHF Servlet :'(
                                                </li> 
                                                <li>
                                                    Do not create a URI in Servlet and defer URI decoding (some kind of lazy decoding: the caller will have to take care of potential exceptions)
                                                </li> 
                                                <li>
                                                    still create an URI, but remove the query string from it, leaving the form parameters as a request property
                                                </li> 
                                            </ul> 
                                            <p style="margin: 10px 0 0 0">My preference goes to #1, clearly. If CHF could respect standards, that would be better.</p> 
                                            <p style="margin: 10px 0 0 0">Note that <tt>&quot;</tt> case is not alone, we already had a similar case in OpenIG (pre-CHF) with an invalid char in the query string (<a href="https://lists.forgerock.org/pipermail/openig/2014-November/000515.html" class="external-link" rel="nofollow" style="color: #3b73af; text-decoration: none">https://lists.forgerock.org/pipermail/openig/2014-November/000515.html</a>).</p>
