        <p style='margin-top:0;margin-bottom:10px;'>We currently have the "_fields" optional property available for READ requests (and maybe QUERY requests?). This acts as a "whitelist" of fields which you would like to see returned in the response body. In addition to this, it would be beneficial to have a "blacklist" of fields you definitely do not want to see returned. This would be helpful when you are storing large amounts of data (likely an encoded binary) in a field in a resource. </p>

<p style='margin-top:0;margin-bottom:10px;'>This is particularly more likely when we start making more use of the facilities available from <a href="https://bugster.forgerock.org/jira/browse/CREST-108" title="Support the ability to update multi-media resource content such as JPEGs"><del>CREST-108</del></a> and <a href="https://bugster.forgerock.org/jira/browse/CREST-85" title="Support the ability to read multi-media resource content such as JPEGs"><del>CREST-85</del></a>. The problem that we are likely to start facing is that as we perform queries or reads to get the "normal" fields back for a particular resource, we will by default also be getting these encoded binary properties. Typically we would rather have a separate request (a la <a href="https://bugster.forgerock.org/jira/browse/CREST-85" title="Support the ability to read multi-media resource content such as JPEGs"><del>CREST-85</del></a>) to read these fields directly (with their proper mime-type and decoded) rather than lumped in with the rest of the data for a record. Having a way to filter these out of the response will allow us to prevent a large amount of wasted data transmission.</p>
