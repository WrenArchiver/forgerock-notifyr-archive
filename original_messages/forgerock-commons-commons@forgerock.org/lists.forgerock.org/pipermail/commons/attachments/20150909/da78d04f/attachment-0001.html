                                            <p style="margin: 10px 0 0 0">As of <a href="https://bugster.forgerock.org/jira/browse/CHF-6" title="Consider merging ClientInfo, ClientInfoContext, and HttpContext" class="issue-link" data-issue-key="CHF-6" style="color: #3b73af; text-decoration: none"></a></p>
                                            <del>
                                                CHF-6
                                            </del> the HttpRequestContext has been split into two parts, an AttributesContext and a SessionContext. In addition, the Principal field has been removed since it was never used in OpenIG and applications should use the SecurityContext instead, although it needs to be moved from CREST to CHF (<a href="https://bugster.forgerock.org/jira/browse/CHF-68" title="Move SecurityContext to CHF" class="issue-link" data-issue-key="CHF-68" style="color: #3b73af; text-decoration: none">CHF-68</a>). 
                                            <p style="margin: 10px 0 0 0">The AttributesContext is a mechanism for transferring transient state between components when processing a single request. Serialization addresses the requirement for the ability to suspend processing of a request until a later date. It makes sense therefore that the attributes should be recovered when processing is resumed.</p> 
                                            <p style="margin: 10px 0 0 0">The SessionContext is a mechanism for maintaining state between components when processing successive requests from the same logical client or end-user. A request that has been suspended can no longer be considered to be in sequence with other requests, so I don't think that it makes sense to serialize and, more importantly, deserialize the SessionContext. Usually the session is used for storing information for the purposes of authorization and auditing, so its content will have already been used in order to populate other contexts, such as the SecurityContext which can be serialized. However, there is still a requirement to serialize the SessionContext as JSON: when it is stored in a JWT session cookie. Therefore, we should ensure that a session's content comprises of JSON compatible objects, even if we leave the responsibility of serialization to the JWT session manager.</p> 
                                            <p style="margin: 10px 0 0 0"><b>Suggested fix</b></p> 
                                            <ul> 
                                                <li>
                                                    AttributesContext should support serialization, so we should store the attributes directly in the context's data. Note that there is a risk of naming collision since all contexts share certain common fields, e.g. &quot;id&quot;, &quot;class&quot;, and &quot;name&quot;. We should prevent manipulation of these core properties, i.e. they should be reserved attribute names
                                                </li> 
                                            </ul> 
                                            <ul> 
                                                <li>
                                                    SessionContext should not have to support serialization, so we should not store the session properties in the context's data. However, care should be taken to ensure that SessionContext is script-friendly, see <a href="https://bugster.forgerock.org/jira/browse/CHF-66" title="Enhance Context to be script-friendly" class="issue-link" data-issue-key="CHF-66" style="color: #3b73af; text-decoration: none">CHF-66</a>.
                                                </li> 
                                            </ul>
