                                            <p style="margin: 10px 0 0 0">Matt, <a href="https://bugster.forgerock.org/jira/secure/ViewProfile.jspa?name=chad.kienle" class="user-hover" rel="chad.kienle" style="color: #3b73af; text-decoration: none">Chad Kienle</a> and I have been working on the implementation of remainingPagedResults within OpenIDM - you can see the initial progress of that work, here: <a href="http://sources.forgerock.org/cru/CR-5643" class="external-link" rel="nofollow" style="color: #3b73af; text-decoration: none">http://sources.forgerock.org/cru/CR-5643</a> . In discussing this, we realized that use-cases for when someone needs to know the remainingPagedResults value seem to always be the same as those that involve the use of the pagedResultsOffset. This realization means that we don't need to introduce a new parameter in the request (as we originally had planned) and instead can model the behavior like so:</p> 
                                            <p style="margin: 10px 0 0 0">1) Assume that there is only ever one type of paging at play - either cookie-based or offset-based. A single query may never have both types of paging.<br /> 2) Determine if there was paging requested at all - this can be detected solely via the presence of the _pageSize parameter.<br /> 3) If paging, determine what sort of paging is requested (cookie or offset). If there is no _pagedResultsCookie or _pagedResultsOffset parameter included in the request, then cookie-based paging is the default. (This is because calculating the cookie is a cheaper process than calculating the total, so there is less risk). If the user includes _pagedResultsOffset in the request, then instead of returning a cookie value the query response includes the remainingPagedResults.<br /> 4) The client knows about the presence of subsequent pages based on either the presence of a non-null cookie value (only present when using cookie-based paging) or a non-zero remainingPagedResults value (when using offset-based paging).</p> 
                                            <p style="margin: 10px 0 0 0">For this scenario to be possible, we have to have a clear way to distinguish the types of paging at step 3, particularly for the initial request. Presently, for the initial request there would be no _pagedResultsCookie or _pagedResultsOffset parameters explicitly defined by the client, and so the sort of paging they want is ambiguous. As stated above, cookie-based should be the default, however there needs to be a way for the user to indicate that they want to use offset-based paging as part of the initial request. This could be accomplished by explicitly including _pagedResultsOffset=0 onto that initial request (the zero indicating that they want to start at the beginning of the result set). The presence of this parameter is enough to indicate the type of paging requested, and thus is able to do the work in step 4.</p> 
                                            <p style="margin: 10px 0 0 0">The problem with this theory is that at present, CREST sets the pagedResultsOffset value to 0 when it isn't explicitly included in the request. This means that there is no way for a query implementation to distinguish between a request that has explicitly indicated that it wants offset-based paging vs. one that didn't indicate either way (which should mean cookie-based). To resolve this ambiguity, I suggest changing the default value for pagedResultsOffset from 0 to -1, which could be used to indicate to the implementation that there was no explicit offset provided (because -1 would be an invalid offset value).</p>
