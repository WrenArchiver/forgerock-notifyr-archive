1 + # API DESCRIPTOR
2 +
3 + ### Description
4 +
5 + RAPID-5
6 +
7 + As both ForgeRock customers and internal developers increasingly favor REST APIs to access the ForgeRock Identity Platform, they all depend on clear descriptions of the API contracts. Today, we maintain descriptions by hand through reverse engineering. We lack ways of ensuring this hand-written documentation is updated when the implementations change, which is an impediment to internal development. Furthermore, our products lack the ability to describe the APIs as they are exposed at runtime in deployment and hence make it harder for community members and customers to use our APIs.
8 +
9 + ### Benefits
10 +
11 + There are multiple benefits of a well-functioning API Descriptor, both for ForgeRock Engineering as well as ForgeRock customers and community members:
12 +
13 + * Enablement of developers to more easily understand and use ForgeRock APIs.
14 + * Ability for developers to test APIs before using them.
15 + * Ability to dynamically generate API documentation.
16 + * Ability to dynamically create user interfaces that are built from API calls.
17 +
18 + ### Scenarios
19 +
20 + * Community members or customer prospects evaluate ForgeRock products for easy of use and interoperability.
21 + * Customers investigate ways to maximize the use of, and potentially extend, the Platform to fit their business needs.
22 + * Prospects and customers test expected behaviour of ForgeRock functionality.
23 + * Developers get access to API descriptions only of those that they are allowed to use.
24 + * Developers get access to SDKs that are automatically generated from the API Descriptor.
25 + * The ForgeRock documentation team generates and maintains API documentation.
26 + * The ForgeRock UI teams develop new interfaces.
27 +
28 + ### Features list
29 +
30 + **Highlights:**
31 +
32 + * A specification that clearly defines how to describe our APIs
33 + * Tools to build and generate API descriptors from our code
34 + * Extensions for Swagger
35 + * Endpoints to filter content for API descriptions
36 + * Endpoints to discover API descriptions
37 + * Automatic generation of API documentation
38 + * A ForgeRock-consistent UI to explore the APIs

56 + ### Customer Feedback
57 +
58 + (Warren:) SDK generation, in as many languages as possible, is the number one reason to implement an API Descriptor. Customers do more and more DevOps work in **Go**, like many others. Most developers do not want to read REST docs. They want to use a library that does it for them.
