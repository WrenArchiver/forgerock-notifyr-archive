1 + # ForgeRock Doc Maven Plugin
2 +
3 + The ForgeRock Doc Maven Plugin builds ForgeRock core documentation
4 + from DocBook XML sources.
5 +
6 + This README describes what the plugin expects of core documentation sources,
7 + and shows how to use some of its key features.
8 +
9 + The ForgeRock Doc Maven Plugin relies on default branding and common content.
10 + The README shows how to use alternative branding and common content.
11 +
12 + _This document covers functionality present in 4.0.0._
13 +
14 +
15 + # About the ForgeRock Doc Maven Plugin
16 +
17 + The Maven plugin centralizes configuration of core documentation,
18 + to ensure that documents are formatted and laid out uniformly.
19 +
20 + The project runs multiple plugin executions:
21 +
22 + 1.  A `pre-site` phase `process` goal to pre-process documents.
23 + 2.  A `pre-site` phase `build` goal to format documents (HTML, PDF, etc.)
24 + 3.  A `site` phase `site` goal to lay out documents in site format
25 + 4.  A `site` phase `release` goal to lay out documents in release format
26 + 5.  A `site` phase `backstage` goal to lay out documents in BackStage format
27 +
28 + With centralized configuration handled by this Maven plugin,
29 + the overall configuration requires at least these arguments:
30 +
31 + *   `<projectName>`: OpenAM, OpenDJ, OpenICF, OpenIDM, OpenIG, and so forth
32 + *   `<projectVersion>`: the version, such as `1.0.0-SNAPSHOT`, or `3.3.1`
33 + *   `<releaseVersion>`: the release version, such as `1.0.0`, or `3.3.1`
34 +
35 + Other features are described in this README.
36 +
37 +
38 + ## Plugin Configuration
39 +
40 + You call the plugin from your `pom.xml` as in the following example.
41 +
42 +     <build>
43 +      <plugins>
44 +       <plugin>
45 +        <groupId>org.forgerock.commons</groupId>
46 +        <artifactId>forgerock-doc-maven-plugin</artifactId>
47 +        <version>${forgerockDocPluginVersion}</version>
48 +        <inherited>false</inherited>
49 +        <configuration>
50 +         <projectName>MyProject</projectName>
51 +         <projectVersion>1.0.0-SNAPSHOT</projectVersion>
52 +         <releaseVersion>1.0.0</releaseVersion>
53 +        </configuration>
54 +        <executions>
55 +         <execution>
56 +          <id>pre-process-doc</id>
57 +          <phase>pre-site</phase>
58 +          <goals>
59 +           <goal>process</goal>
60 +          </goals>
61 +         </execution>
62 +         <execution>
63 +          <id>build-doc</id>
64 +          <phase>pre-site</phase>
65 +          <goals>
66 +           <goal>build</goal>
67 +          </goals>
68 +         </execution>
69 +         <execution>
70 +          <id>layout-site</id>
71 +          <phase>site</phase>
72 +          <goals>
73 +           <goal>site</goal>
74 +          </goals>
75 +         </execution>
76 +         <execution>
77 +          <id>layout-release</id>
78 +          <phase>site</phase>
79 +          <goals>
80 +           <goal>release</goal>
81 +          </goals>
82 +         </execution>
83 +         <execution>
84 +          <id>layout-backstage</id>
85 +          <phase>site</phase>
86 +          <goals>
87 +           <goal>backstage</goal>
88 +          </goals>
89 +         </execution>
90 +        </executions>
91 +       </plugin>
92 +      </plugins>
93 +     </build>
94 +
95 + In the example above, `<inherited>false</inherited>` means
96 + only use this plugin configuration for this project or module,
97 + not recursively for all modules that are children of this project.
98 +
99 + Unless you want to run the plugin recursively, set `<inherited>false</inherited>`.

101 + ## Alternative Plugin Configuration
102 +
103 + When documentation is the sole subject of the POM,
104 + either because the project is only for documentation
105 + or because the plugin is used in a module that is only for documentation,
106 + you can use `docs` project packaging.
107 + The `docs` packaging redefines default lifecycle goals
108 + to run `forgerock-doc-maven-plugin` without specifying executions:
109 +

115 +             <plugins>

119 +               <version>3.2.0</version>
120 +               <extensions>true</extensions>
121 +               <configuration>
122 +                 <projectName>Test</projectName>
123 +                 <projectVersion>1.0.0-SNAPSHOT</projectVersion>

169 + ## Plugin Output
170 +
171 + When you run the `pre-site` phase `build` goal,
172 + the plugin builds the HTML, PDF, etc.,
173 + which you find under `${project.build.directory}/docbkx`,
174 + which is usually `target/docbkx`.
175 +
176 + When you run the `site` phase `site` goal,
177 + the plugin copies the documents it built during the `pre-site` phase
178 + under `${project.build.directory}/site/doc` as expected for a Maven project site.
179 +
180 + The plugin adds an `index.html` in the `site/doc` directory.
181 + That `index.html` file redirects browsers
182 + to `http://project.forgerock.org/docs.html`,
183 + so be sure to add a `docs.html` to your Maven site.
184 +
185 + When you run the `site` phase `release` goal,
186 + the plugin lays out copies of HTML and PDF
187 + under `${project.build.directory}/release/releaseVersion`,
188 + where `releaseVersion` is the release version set in the configuration.
189 +
190 + The files in `release/releaseVersion` are those suitable for publication.
191 +
192 +
193 + ## Source Layout Requirements
194 +
195 + The plugin assumes that all of your DocBook XML documents
196 + are found under `src/main/docbkx/`.
197 +
198 + Documents are expected to be found in folders under that path,
199 + where the folder name is a lowercase version of the document name,
200 + such as `release-notes`, `admin-guide`, `reference`, or similar.
201 +
202 + Furthermore, all documents have the same file name
203 + for the file containing the top-level document element, by default `index.xml`.
204 +
205 + The plugin expects all images in an `images` folder inside the document folder.
206 +
207 + An example project layout looks like this:
208 +
209 +      src/main/docbkx/
210 +       legal.xml
211 +       admin-guide/
212 +        images/
213 +        index.xml
214 +        ...other files...
215 +       dev-guide/
216 +        images/
217 +        index.xml
218 +        ...other files...
219 +       install-guide/
220 +        images/
221 +        index.xml
222 +        ...other files...
223 +       reference/
224 +        images/
225 +        index.xml
226 +        ...other files...
227 +       release-notes/
228 +        images/
229 +        index.xml
230 +        ...other files...
231 +       shared/
232 +        sec-accessing-doc-online.xml
233 +        sec-formatting-conventions.xml
234 +        sec-interface-stability.xml
235 +        sec-joining-the-community.xml
236 +        sec-release-levels.xml
237 +        ...other files...
238 +
239 + During the build,
240 + the plugin makes a copy of the original sources under the build directory.
241 + It then works on the copy, rather than the original.

244 + # ForgeRock Doc Maven Plugin Features

248 + ## Generating a Basic Documentation Project
249 +
250 + The `forgerock-doc-maven-archetype` can be used 
251 + to generate a basic documentation project.
252 +
253 + To generate files for MyProject for example:
254 +
255 +     mvn archetype:generate                                  \
256 +       -DarchetypeRepository=http://maven.forgerock.org/repo \
257 +       -DarchetypeGroupId=commons.forgerock.org              \
258 +       -DarchetypeArtifactId=forgerock-doc-maven-archetype   \
259 +       -DarchetypeVersion=3.0.0                              \
260 +       -DgroupId=projectGroupId                              \
261 +       -DartifactId=projectArtifactId                        \
262 +       -DgoogleAnalyticsId=UA-xxxxxxxx-x                     \
263 +       -Dname=MyProject                                      \
264 +       -Dversion=1.0.0-SNAPSHOT
265 +
266 + In general, after creating your new documentation set by using the archetype,
267 + go through each of the XML files to determine what you must change.
268 + Your documentation set might not even have an _Admin Guide_ or _Reference_,
269 + though you can still use the files as templates.

272 + ## Using Variables in Documents
273 +
274 + The `pre-site` phase `build` goal applies the Maven resource filtering.
275 +  
276 + This means you can use Maven properties to add variables to your documentation.
277 +
278 + For example, say you have a property defined in your POM:
279 +
280 +     <properties>
281 +       <myUrl>http://docs.forgerock.org/</myUrl>
282 +       ...
283 +     </properties>
284 +
285 + During the `pre-site` phase `build` goal,
286 + the plugin replaces `${myUrl}`
287 + in the build copy of the documents with `http://docs.forgerock.org`.
288 +
289 + This allows you to use Maven properties in attribute values,
290 + such as `xlink:href="${myURL}"`,
291 + whereas the form using a processing instruction,
292 + `xlink:href="<?eval ${myURL}"?>`,
293 + is not valid XML.
294 +

296 + ## Building Release Documentation
297 +
298 + The `site` phase `release` goal prepares documents for publication.
299 +
300 + You might choose to include a release profile in your project.
301 + Or you might choose to call the release goal by hand.
302 +
303 + When you call the `release` goal, be sure to
304 + set the appropriate dates, turn off draft mode, and override the Google Analytics ID
305 + as shown in the following example.
306 +
307 +      mvn \
308 +      -DisDraftMode=no  \
309 +      -D"gaId=UA-23412190-14" \
310 +      -D"releaseDate=Software release date: January 1, 1970" \
311 +      -D"pubDate=Publication date: December 31, 1969" \
312 +      -DbuildReleaseZip=true \
313 +      clean site org.forgerock.commons:forgerock-doc-maven-plugin:release
314 +
315 + If the plugin configuration is not inherited,
316 + then also set `-N` (`--non-recursive`) for the release goal.
317 + Run the `site` goal separately if it must be recursive
318 + (because you build Javadoc during the `site` goal for example).
319 +
320 + Appropriate dates should be included in the documents to publish.
321 + * The `releaseDate` indicates the date the software was released.
322 + * The `pubDate` indicates the date you published the documentation.
323 +
324 + To build a .zip of the release documentation,
325 + you can set `-DbuildReleaseZip=true` as shown in the above example.
326 + The file, `projectName-releaseVersion-docs.zip`,
327 + can be found after the build in the project build directory.
328 + When unzipped, it unpacks the documentation for the release
329 + under `projectName/releaseVersion/`.

332 + ## Building BackStage Layout Documentation
333 +
334 + The `backstage` goal allows you to generate docs in a layout
335 + suitable for ForgeRock BackStage.
336 +
337 + If the documentation set includes pre-built artifacts, such as Javadoc,
338 + then you must specify those artifacts in the configuration.
339 + For example:
340 +
341 +      <artifactItems>
342 +       <artifactItem>
343 +        <groupId>org.forgerock.commons</groupId>
344 +        <artifactId>forgerock-doc-maven-plugin</artifactId>
345 +        <version>${project.version}</version>
346 +        <classifier>javadoc</classifier>
347 +        <outputDirectory>javadoc</outputDirectory>
348 +        <title>ForgeRock Doc Maven Plugin Javadoc</title>
349 +       </artifactItem>
350 +      </artifactItems>
351 +      
352 + To help BackStage sort the document titles into the desired order,
353 + also specify `<documents>` by their XML ID as shown in the following example.
354 + Otherwise BackStage sorts the documents alphabetically:
355 +
356 +     <documents>
357 +      <document>release-notes</document>
358 +      <document>admin-guide</document>
359 +      <document>reference</document>
360 +      <document>javadoc</document>
361 +     </documents>
362 +
363 + In the example, notice that the Javadoc retrieved from an artifact
364 + is specified by its output directory.
365 +
366 + BackStage layout is as follows:
367 +
368 + * `apidocs/` contains folders of any generated HTML-based documentation,
369 +    such as Javadoc, that is not built from normal documentation sources,
370 +    including a `meta.json` file inside each folder
371 +    to specify the name of the document.
372 +    The artifacts must be specified in the configuration.
373 +    An example `meta.json` file looks like this:
374 +
375 +
376 +     {
377 +         "title": "OpenAM Javadoc"
378 +     }
379 +
380 + * `docbook/` contains the pre-processed DocBook XML sources
381 +   suitable for formatting by a separate program.
382 +
383 + * `docset.json` specifies meta information about the documentation set.
384 +   The optional `documents` array specifies intended order of documents.
385 +   For example:
386 +
387 +
388 +     {
389 +         "product": "OpenAM",
390 +         "version": "12.0.0",
391 +         "language": "en",
392 +         "released": "2014-12-17",
393 +         "documents": ["release-notes", "admin-guide", "reference", "javadoc"]
394 +     }
395 +
396 + * `pdf/` contains PDF files corresponding to the DocBook XML sources,
397 +   named as `<Product-from-docset-json>-<Version>-<Doc-name>.pdf`.
398 +
399 + By default the product name is from `<projectName>`.
400 +      
401 + This goal also builds an artifact of the PDF files,
402 + which contains all the files at the root of the JAR.
403 + The artifact has the classifier `pdf`.
404 + Its group ID, artifact ID, and version match those of the project.

407 + ## Generating Single-Chapter Output
408 +
409 + By default, the plugin generates output for each document
410 + whose root is named `index.xml`.
411 +
412 + You can change this by setting `documentSrcName` when you run Maven.
413 +
414 + The following example produces pre-site output
415 + only for a chapter named `chap-one.xml`.
416 +
417 +     mvn -DdocumentSrcName=chap-one.xml clean pre-site
418 +
419 +
420 + ## Selecting Output Formats
421 +
422 + Specify the type of outputs you want using `<formats>`.
423 + On the command line, use `-Dformats` with a comma-separated list.
424 +
425 + Supported formats include:
426 + `bootstrap`, `epub`, `html`, `man`, `pdf`, `rtf`, `webhelp`, `xhtml5`.
427 +
428 + If you do not specify any formats, the default output formats are
429 + `bootstrap` and `pdf`.
430 +
431 + Release builds only include HTML and `pdf`.
432 +
433 + The following command generates only PDF output for your single chapter.
434 +
435 +     mvn -DdocumentSrcName=chap-one.xml -Dformats=pdf clean pre-site
436 +
437 + The following command generates only man pages and XHTML5 output.
438 +
439 +     mvn -Dformats=man,xhtml5 clean pre-site
440 +
441 +
442 + ## Checking Links
443 +
444 + By default, the plugin checks links found in the DocBook XML source,
445 + including Olinks.
446 +
447 + You can find errors in `${project.build.directory}/docbkx/linktester.err`
448 + after the `site` phase `site` goal runs successfully.
449 +
450 + To skip the checks for external URLs, pass `-DskipLinkCheck=true` to Maven.
451 +
452 +     mvn -DskipLinkCheck=true clean site
453 +
454 + To skip only some external URLs, use the `<skipUrlPatterns>` configuration list.
455 +
456 +     <skipUrlPatterns>
457 +      <skipUrlPattern>^http(s)?://my-host.*$</skipUrlPattern>
458 +      <skipUrlPattern>^http(s)?://.*siroe.*$</skipUrlPattern>
459 +     </skipUrlPatterns>
460 +
461 + This capability is provided by Peter Major's
462 + [linktester](https://github.com/aldaris/docbook-linktester) plugin.
463 + For details on skipping URLs by pattern, see the documentation for his plugin.

466 + ## Adding Asciidoc Books
467 +
468 + Asciidoc books can now be added alongside DocBook XML books.
469 + The Asciidoc format books are converted to DocBook XML during pre-processing.
470 + The current implementation does not support mixing
471 + Asciidoc and DocBook XML sources for the same book.
472 +
473 + Lay out your Asciidoc book sources in the same way you lay out DocBook sources,
474 + under `src/main/asciidoc` by default.
475 + Use `<asciidocSourceDirectory>` if necessary to modify the source location.

478 + ## Handling PNG Images
479 +
480 + Getting screenshots and other images to look okay in PDF can be a hassle.
481 + The plugin therefore adjusts images to make large PNG images fit in the page,
482 + and adjusts dots-per-inch on PNG images to make them look okay in print.
483 +
484 + **Note: Do capture screenshots at 72 DPI. Retina displays can default to 144.**
485 +

487 + ## Using Syntax Highlighting in HTML
488 +
489 + Uses [SyntaxHighlighter](http://alexgorbatchev.com/SyntaxHighlighter/),

497 +      csv            csv                 shBrushCsv.js
498 +      html           html                shBrushXml.js
499 +      http           http                shBrushHttp.js
500 +      ini            ini                 shBrushProperties.js
501 +      java           java                shBrushJava.js
502 +      javascript     javascript          shBrushJScript.js

523 +
524 + During the pre-processing phase the build plugin

643 + ## Citing Text Files
644 +
645 + The build now uses the `xcite-maven-plugin` (XCite)
646 + to allow citations from any text file.
647 +
648 + XCite works by replacing citation strings with quotes.
649 +
650 +     Hello world.
651 +     This is a file that quotes `/path/to/script.sh.`
652 +     
653 +     Here comes the quote:
654 +     
655 +     [/path/to/script.sh:# start:# end]
656 +     
657 +     And here's the rest of the file.
658 +
659 + In the example shown, the citation string is
660 + `[/path/to/script.sh:# start:# end]`.
661 +
662 + Suppose `/path/to/script.sh` contains the following content.
663 +
664 +      #!/bin/bash
665 +      
666 +      # start
667 +      wall <<EOM
668 +          Hello world
669 +      EOM
670 +      # end
671 +      
672 +      exit
673 +
674 + Then running XCite on the file results in the following output.
675 +
676 +     Hello world.
677 +     This is a file that quotes `/path/to/script.sh.`
678 +     
679 +     Here comes the quote:
680 +     
681 +      wall <<EOM
682 +          Hello world
683 +      EOM
684 +     
685 +     And here's the rest of the file.
686 +
687 + Notice that the citation has the following form,
688 + inspired by [JCite](http://www.arrenbrecht.ch/jcite/):
689 +
690 +     [path:start-marker:end-marker]
691 +
692 + * The `path` is the path to the file to quote.
693 + * The `start-marker` is the string marking the start of the quote.
694 + * The `end-marker` is the string marking the end of the quote.
695 +
696 + The `path` is required, and must reference a readable file (assumed to be text).
697 +
698 + If you omit the `start-marker`, XCite includes the entire file.
699 +
700 + If you include the `start-marker` but omit the `end-marker`,
701 + XCite assumes that the `start-marker` and `end-marker` are identical.
702 +
703 + Neither `path`, nor `start-marker`, nor `end-marker` can contain the delimiter.
704 + The delimiter is `:` by default, optionally `%` instead.
705 +
706 + The `start-marker` and `end-marker` are not part of the quote.
707 + The `start-marker` and `end-marker` must be either
708 + on separate lines from the quote,
709 + or on the same line as the (single-line) quote.
710 + When the `start-marker` and `end-marker` are on separate lines from the quote,
711 + the lines containing the `start-marker` and `end-marker`
712 + are not part of the quote.
713 +
714 + Suppose `file.txt` contains the following lines:
715 +
716 +     (start)XCite can recognize this as a quote.(end)
717 +
718 +     // The marker is in this line.
719 +     XCite can recognize this as a quote.
720 +     // The marker is in this line.
721 +
722 +     // WRONG
723 +     XCite does not recognize this as a quote. // WRONG
724 +
725 + * The citation string `[file.txt:start:end]` resolves to
726 +   `XCite can recognize this as a quote.`
727 + * The citation string `[file.txt%marker]` resolves to
728 +   `XCite can recognize this as a quote.`
729 + * The citation string `[file.txt:WRONG]` resolves to an empty string.
730 +
731 + Although XCite does resolve citations recursively, it does not check for loops.
732 + A loop occurs when a quote includes a citation quoting the original citation.
733 +
734 + If you create a citation loop,
735 + XCite processes it recursively until it runs out of memory.
