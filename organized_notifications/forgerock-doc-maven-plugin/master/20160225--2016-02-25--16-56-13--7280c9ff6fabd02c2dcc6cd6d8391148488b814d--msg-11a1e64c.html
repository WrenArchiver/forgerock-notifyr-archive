1 + # ForgeRock Doc Maven Plugin
2 +
3 + The ForgeRock Doc Maven Plugin builds ForgeRock core documentation
4 + from DocBook XML sources.
5 +
6 + This README describes what the plugin expects of core documentation sources,
7 + and shows how to use some of its key features.
8 +
9 + The ForgeRock Doc Maven Plugin relies on default branding and common content.
10 + The README shows how to use alternative branding and common content.
11 +
12 + _This document covers functionality present in 3.1.0._
13 +
14 +
15 + # About the ForgeRock Doc Maven Plugin
16 +
17 + The Maven plugin centralizes configuration of core documentation,
18 + to ensure that documents are formatted and laid out uniformly.
19 +
20 + The project runs multiple plugin executions:
21 +
22 + 1.  A `pre-site` phase `process` goal to pre-process documents.
23 + 2.  A `pre-site` phase `build` goal to format documents (HTML, PDF, etc.)
24 + 3.  A `site` phase `site` goal to lay out documents in site format
25 + 4.  A `site` phase `release` goal to lay out documents in release format
26 + 5.  A `site` phase `backstage` goal to lay out documents in BackStage format
27 +
28 + With centralized configuration handled by this Maven plugin,
29 + the overall configuration requires at least these arguments:
30 +
31 + *   `<projectName>`: OpenAM, OpenDJ, OpenICF, OpenIDM, OpenIG, and so forth
32 + *   `<projectVersion>`: the version, such as `1.0.0-SNAPSHOT`, or `3.3.1`
33 + *   `<releaseVersion>`: the release version, such as `1.0.0`, or `3.3.1`
34 +
35 + Other features are described in this README.
36 +
37 +
38 + ## Plugin Configuration
39 +
40 + You call the plugin from your `pom.xml` as in the following example.
41 +
42 +     <build>
43 +      <plugins>
44 +       <plugin>
45 +        <groupId>org.forgerock.commons</groupId>
46 +        <artifactId>forgerock-doc-maven-plugin</artifactId>
47 +        <version>${forgerockDocPluginVersion}</version>
48 +        <inherited>false</inherited>
49 +        <configuration>
50 +         <projectName>MyProject</projectName>
51 +         <projectVersion>1.0.0-SNAPSHOT</projectVersion>
52 +         <releaseVersion>1.0.0</releaseVersion>
53 +        </configuration>
54 +        <executions>
55 +         <execution>
56 +          <id>pre-process-doc</id>
57 +          <phase>pre-site</phase>
58 +          <goals>
59 +           <goal>process</goal>
60 +          </goals>
61 +         </execution>
62 +         <execution>
63 +          <id>build-doc</id>
64 +          <phase>pre-site</phase>
65 +          <goals>
66 +           <goal>build</goal>
67 +          </goals>
68 +         </execution>
69 +         <execution>
70 +          <id>layout-site</id>
71 +          <phase>site</phase>
72 +          <goals>
73 +           <goal>site</goal>
74 +          </goals>
75 +         </execution>
76 +         <execution>
77 +          <id>layout-release</id>
78 +          <phase>site</phase>
79 +          <goals>
80 +           <goal>release</goal>
81 +          </goals>
82 +         </execution>
83 +         <execution>
84 +          <id>layout-backstage</id>
85 +          <phase>site</phase>
86 +          <goals>
87 +           <goal>backstage</goal>
88 +          </goals>
89 +         </execution>
90 +        </executions>
91 +       </plugin>
92 +      </plugins>
93 +     </build>
94 +
95 + In the example above, `<inherited>false</inherited>` means
96 + only use this plugin configuration for this project or module,
97 + not recursively for all modules that are children of this project.
98 +
99 + Unless you want to run the plugin recursively, set `<inherited>false</inherited>`.

101 + ## Plugin Output
102 +
103 + When you run the `pre-site` phase `build` goal,
104 + the plugin builds the HTML, PDF, etc.,
105 + which you find under `${project.build.directory}/docbkx`,
106 + which is usually `target/docbkx`.
107 +
108 + When you run the `site` phase `site` goal,
109 + the plugin copies the documents it built during the `pre-site` phase
110 + under `${project.build.directory}/site/doc` as expected for a Maven project site.
111 +
112 + The plugin adds an `index.html` in the `site/doc` directory.
113 + That `index.html` file redirects browsers
114 + to `http://project.forgerock.org/docs.html`,
115 + so be sure to add a `docs.html` to your Maven site.
116 +
117 + When you run the `site` phase `release` goal,
118 + the plugin lays out copies of HTML and PDF
119 + under `${project.build.directory}/release/releaseVersion`,
120 + where `releaseVersion` is the release version set in the configuration.
121 +
122 + The files in `release/releaseVersion` are those suitable for publication.
123 +
124 +
125 + ## Source Layout Requirements
126 +
127 + The plugin assumes that all of your DocBook XML documents
128 + are found under `src/main/docbkx/`.
129 +
130 + Documents are expected to be found in folders under that path,
131 + where the folder name is a lowercase version of the document name,
132 + such as `release-notes`, `admin-guide`, `reference`, or similar.
133 +
134 + Furthermore, all documents have the same file name
135 + for the file containing the top-level document element, by default `index.xml`.
136 +
137 + The plugin expects all images in an `images` folder inside the document folder.
138 +
139 + An example project layout looks like this:
140 +
141 +      src/main/docbkx/
142 +       legal.xml
143 +       admin-guide/
144 +        images/
145 +        index.xml
146 +        ...other files...
147 +       dev-guide/
148 +        images/
149 +        index.xml
150 +        ...other files...
151 +       install-guide/
152 +        images/
153 +        index.xml
154 +        ...other files...
155 +       reference/
156 +        images/
157 +        index.xml
158 +        ...other files...
159 +       release-notes/
160 +        images/
161 +        index.xml
162 +        ...other files...
163 +       shared/
164 +        sec-accessing-doc-online.xml
165 +        sec-formatting-conventions.xml
166 +        sec-interface-stability.xml
167 +        sec-joining-the-community.xml
168 +        sec-release-levels.xml
169 +        ...other files...
170 +
171 + During the build,
172 + the plugin makes a copy of the original sources under the build directory.
173 + It then works on the copy, rather than the original.

180 + ## Generating a Basic Documentation Project
181 +
182 + The `forgerock-doc-maven-archetype` can be used 
183 + to generate a basic documentation project.
184 +
185 + To generate files for MyProject for example:
186 +
187 +     mvn archetype:generate                                  \
188 +       -DarchetypeRepository=http://maven.forgerock.org/repo \
189 +       -DarchetypeGroupId=commons.forgerock.org              \
190 +       -DarchetypeArtifactId=forgerock-doc-maven-archetype   \
191 +       -DarchetypeVersion=3.0.0                              \
192 +       -DgroupId=projectGroupId                              \
193 +       -DartifactId=projectArtifactId                        \
194 +       -DgoogleAnalyticsId=UA-xxxxxxxx-x                     \
195 +       -Dname=MyProject                                      \
196 +       -Dversion=1.0.0-SNAPSHOT
197 +
198 + In general, after creating your new documentation set by using the archetype,
199 + go through each of the XML files to determine what you must change.
200 + Your documentation set might not even have an _Admin Guide_ or _Reference_,
201 + though you can still use the files as templates.

204 + ## Using Variables in Documents
205 +
206 + The `pre-site` phase `build` goal applies the Maven resource filtering.
207 +  
208 + This means you can use Maven properties to add variables to your documentation.
209 +
210 + For example, say you have a property defined in your POM:
211 +
212 +     <properties>
213 +       <myUrl>http://docs.forgerock.org/</myUrl>
214 +       ...
215 +     </properties>
216 +
217 + During the `pre-site` phase `build` goal,
218 + the plugin replaces `${myUrl}`
219 + in the build copy of the documents with `http://docs.forgerock.org`.
220 +
221 + This allows you to use Maven properties in attribute values,
222 + such as `xlink:href="${myURL}"`,
223 + whereas the form using a processing instruction,
224 + `xlink:href="<?eval ${myURL}"?>`,
225 + is not valid XML.

228 + ## Building Release Documentation
229 +
230 + The `site` phase `release` goal prepares documents for publication.
231 +
232 + You might choose to include a release profile in your project.
233 + Or you might choose to call the release goal by hand.
234 +
235 + When you call the `release` goal, be sure to
236 + set the appropriate dates, turn off draft mode, and override the Google Analytics ID
237 + as shown in the following example.
238 +
239 +      mvn \
240 +      -DisDraftMode=no  \
241 +      -D"gaId=UA-23412190-14" \
242 +      -D"releaseDate=Software release date: January 1, 1970" \
243 +      -D"pubDate=Publication date: December 31, 1969" \
244 +      -DbuildReleaseZip=true \
245 +      clean site org.forgerock.commons:forgerock-doc-maven-plugin:release
246 +
247 + If the plugin configuration is not inherited,
248 + then also set `-N` (`--non-recursive`) for the release goal.
249 + Run the `site` goal separately if it must be recursive
250 + (because you build Javadoc during the `site` goal for example).
251 +
252 + Appropriate dates should be included in the documents to publish.
253 + * The `releaseDate` indicates the date the software was released.
254 + * The `pubDate` indicates the date you published the documentation.
255 +
256 + To build a .zip of the release documentation,
257 + you can set `-DbuildReleaseZip=true` as shown in the above example.
258 + The file, `projectName-releaseVersion-docs.zip`,
259 + can be found after the build in the project build directory.
260 + When unzipped, it unpacks the documentation for the release
261 + under `projectName/releaseVersion/`.

264 + ## Building BackStage Layout Documentation
265 +
266 + The `backstage` goal allows you to generate docs in a layout
267 + suitable for ForgeRock BackStage.
268 +
269 + If the documentation set includes pre-built artifacts, such as Javadoc,
270 + then you must specify those artifacts in the configuration.
271 + For example:
272 +
273 +      <artifactItems>
274 +       <artifactItem>
275 +        <groupId>org.forgerock.commons</groupId>
276 +        <artifactId>forgerock-doc-maven-plugin</artifactId>
277 +        <version>${project.version}</version>
278 +        <classifier>javadoc</classifier>
279 +        <outputDirectory>javadoc</outputDirectory>
280 +        <title>ForgeRock Doc Maven Plugin Javadoc</title>
281 +       </artifactItem>
282 +      </artifactItems>
283 +
284 + BackStage layout is as follows:
285 +
286 + * `apidocs/` contains folders of any generated HTML-based documentation,
287 +    such as Javadoc, that is not built from normal documentation sources,
288 +    including a `meta.json` file inside each folder
289 +    to specify the name of the document.
290 +    The artifacts must be specified in the configuration.
291 +    An example `meta.json` file looks like this:
292 +
293 +
294 +     {
295 +         "title": "OpenAM Javadoc"
296 +     }
297 +
298 + * `docbook/` contains the pre-processed DocBook XML sources
299 +   suitable for formatting by a separate program.
300 +
301 + * `docset.json` specifies meta information about the documentation set.
302 +   For example:
303 +
304 +
305 +     {
306 +         "product": "OpenAM",
307 +         "version": "12.0.0",
308 +         "language": "en",
309 +         "released": "2014-12-17"
310 +     }
311 +
312 + * `pdf/` contains PDF files corresponding to the DocBook XML sources,
313 +   named as `<Product-from-docset-json>-<Version>-<Doc-name>.pdf`.
314 +
315 + By default the product name is from `<projectName>`.
316 +      
317 + This goal also builds an artifact of the PDF files,
318 + which contains all the files at the root of the JAR.
319 + The artifact has the classifier `pdf`.
320 + Its group ID, artifact ID, and version match those of the project.

323 + ## Generating Single-Chapter Output
324 +
325 + By default, the plugin generates output for each document
326 + whose root is named `index.xml`.
327 +
328 + You can change this by setting `documentSrcName` when you run Maven.
329 +
330 + The following example produces pre-site output
331 + only for a chapter named `chap-one.xml`.
332 +
333 +     mvn -DdocumentSrcName=chap-one.xml clean pre-site
334 +
335 +
336 + ## Selecting Output Formats
337 +
338 + Specify the type of outputs you want using `<formats>`.
339 + On the command line, use `-Dformats` with a comma-separated list.
340 +
341 + Supported formats include:
342 + `bootstrap`, `epub`, `html`, `man`, `pdf`, `rtf`, `webhelp`, `xhtml5`.
343 +
344 + If you do not specify any formats, the default output formats are
345 + `bootstrap` and `pdf`.
346 +
347 + Release builds only include HTML and `pdf`.
348 +
349 + The following command generates only PDF output for your single chapter.
350 +
351 +     mvn -DdocumentSrcName=chap-one.xml -Dformats=pdf clean pre-site
352 +
353 + The following command generates only man pages and XHTML5 output.
354 +
355 +     mvn -Dformats=man,xhtml5 clean pre-site
356 +

358 + ## Checking Links
359 +
360 + By default, the plugin checks links found in the DocBook XML source,
361 + including Olinks.
362 +
363 + You can find errors in `${project.build.directory}/docbkx/linktester.err`
364 + after the `site` phase `site` goal runs successfully.
365 +
366 + To skip the checks for external URLs, pass `-DskipLinkCheck=true` to Maven.
367 +
368 +     mvn -DskipLinkCheck=true clean site
369 +
370 + To skip only some external URLs, use the `<skipUrlPatterns>` configuration list.
371 +
372 +     <skipUrlPatterns>
373 +      <skipUrlPattern>^http(s)?://my-host.*$</skipUrlPattern>
374 +      <skipUrlPattern>^http(s)?://.*siroe.*$</skipUrlPattern>
375 +     </skipUrlPatterns>
376 +
377 + This capability is provided by Peter Major's
378 + [linktester](https://github.com/aldaris/docbook-linktester) plugin.
379 + For details on skipping URLs by pattern, see the documentation for his plugin.

382 + ## Adding Asciidoc Books
383 +
384 + Asciidoc books can now be added alongside DocBook XML books.
385 + The Asciidoc format books are converted to DocBook XML during pre-processing.
386 + The current implementation does not support mixing
387 + Asciidoc and DocBook XML sources for the same book.
388 +
389 + Lay out your Asciidoc book sources in the same way you lay out DocBook sources,
390 + under `src/main/asciidoc` by default.
391 + Use `<asciidocSourceDirectory>` if necessary to modify the source location.
392 +
393 +
394 + ## Handling PNG Images
395 +
396 + Getting screenshots and other images to look okay in PDF can be a hassle.
397 + The plugin therefore adjusts images to make large PNG images fit in the page,
398 + and adjusts dots-per-inch on PNG images to make them look okay in print.
399 +
400 + **Note: Do capture screenshots at 72 DPI. Retina displays can default to 144.**
401 +

403 + ## Using Syntax Highlighting in HTML
404 +
405 + Uses [SyntaxHighlighter](http://alexgorbatchev.com/SyntaxHighlighter/),
406 + rather than DocBook's syntax highlighting capabilities for HTML output.
407 +
408 + The highlighting operates only inside `<programlisting>`.
409 +
410 +      Source         SyntaxHighlighter   Brush Name
411 +      ---            ---                 ---
412 +      aci            aci                 shBrushAci.js
413 +      csv            csv                 shBrushCsv.js
414 +      html           html                shBrushXml.js
415 +      http           http                shBrushHttp.js
416 +      ini            ini                 shBrushProperties.js
417 +      java           java                shBrushJava.js
418 +      javascript     javascript          shBrushJScript.js
419 +      ldif           ldif                shBrushLDIF.js
420 +      none           plain               shBrushPlain.js
421 +      shell          shell               shBrushBash.js
422 +      xml            xml                 shBrushXml.js
423 +
424 + Brush support for `aci`, `csv`, `http`, `ini`, and `ldif` is provided by
425 + [a fork of SyntaxHighlighter](https://github.com/markcraig/SyntaxHighlighter).
426 +
427 + To set the language for syntax highlighting use the `language` attribute
428 + on the `<programlisting>` element, as in the following example:
429 +
430 +     <programlisting language="java">
431 +     class Test {
432 +         public static void main(String [] args)  {
433 +             System.out.println("This is a program listing.");
434 +         }
435 +     }
436 +     </programlisting>

438 + ## Using Conditional Text
439 +
440 + During the pre-processing phase the build plugin
441 + makes it possible to use conditional text with
442 + [DocBook profiling](http://www.sagehill.net/docbookxsl/Profiling.html) attributes.
443 +
444 + In the source, set profiling attributes on elements to identify their profiles:
445 +
446 +     <para os="linux">This is about Linux.</para>
447 +     <para os="windows">This is about Windows.</para>
448 +     <para condition="local-db">Include if local-db is delivered.</para>
449 +
450 + In the plugin configuration, specify the inclusions and exclusions.
451 + For example, to include the paragraphs about Linux and Windows,
452 + and to exclude the local-db paragraph:
453 +
454 +     <inclusions>
455 +       <os>linux windows</os>
456 +     </inclusions>
457 +     <exclusions>
458 +       <condition>local-db</condition>
459 +     </exclusions>
460 +     
461 + Notice that multiple values for profile attributes are separated by spaces.
462 +     
463 + After pre-processing, the source looks like this:
464 +
465 +     <para os="linux">This is about Linux.</para>
466 +     <para os="windows">This is about Windows.</para>
467 +     
468 + For the list of supported profiling attributes,
469 + see <http://www.sagehill.net/docbookxsl/Profiling.html#ProfilingAttribs>.

472 + ## Building Documentation from Pre-Processed Sources
473 +
474 + When building output directly from pre-processed sources,
475 + use the Maven dependency plugin to retrieve and unpack the sources
476 + before you call this plugin.
477 +
478 +       <plugin>
479 +         <groupId>org.apache.maven.plugins</groupId>
480 +         <artifactId>maven-dependency-plugin</artifactId>
481 +         <executions>
482 +           <execution>
483 +             <goals>
484 +               <goal>unpack</goal>
485 +             </goals>
486 +             <phase>pre-site</phase>
487 +             <configuration>
488 +               <artifactItems>
489 +                 <artifactItem>
490 +                   <groupId>${myGroupId}</groupId>
491 +                   <artifactId>${myArtifactId}</artifactId>
492 +                   <version>${myVersion}</version>
493 +                   <classifier>doc-sources</classifier>
494 +                   <outputDirectory>${project.build.directory}/db-src</outputDirectory>
495 +                 </artifactItem>
496 +               </artifactItems>
497 +             </configuration>
498 +           </execution>
499 +         </executions>
500 +       </plugin>

545 + ## Citing Text Files
546 +
547 + The build now uses the `xcite-maven-plugin` (XCite)
548 + to allow citations from any text file.
549 +
550 + XCite works by replacing citation strings with quotes.
551 +
552 +     Hello world.
553 +     This is a file that quotes `/path/to/script.sh.`
554 +     
555 +     Here comes the quote:
556 +     
557 +     [/path/to/script.sh:# start:# end]
558 +     
559 +     And here's the rest of the file.
560 +
561 + In the example shown, the citation string is
562 + `[/path/to/script.sh:# start:# end]`.
563 +
564 + Suppose `/path/to/script.sh` contains the following content.
565 +
566 +      #!/bin/bash
567 +      
568 +      # start
569 +      wall <<EOM
570 +          Hello world
571 +      EOM
572 +      # end
573 +      
574 +      exit
575 +
576 + Then running XCite on the file results in the following output.
577 +
578 +     Hello world.
579 +     This is a file that quotes `/path/to/script.sh.`
580 +     
581 +     Here comes the quote:
582 +     
583 +      wall <<EOM
584 +          Hello world
585 +      EOM
586 +     
587 +     And here's the rest of the file.
588 +
589 + Notice that the citation has the following form,
590 + inspired by [JCite](http://www.arrenbrecht.ch/jcite/):
591 +
592 +     [path:start-marker:end-marker]
593 +
594 + * The `path` is the path to the file to quote.
595 + * The `start-marker` is the string marking the start of the quote.
596 + * The `end-marker` is the string marking the end of the quote.
597 +
598 + The `path` is required, and must reference a readable file (assumed to be text).
599 +
600 + If you omit the `start-marker`, XCite includes the entire file.
601 +
602 + If you include the `start-marker` but omit the `end-marker`,
603 + XCite assumes that the `start-marker` and `end-marker` are identical.
604 +
605 + Neither `path`, nor `start-marker`, nor `end-marker` can contain the delimiter.
606 + The delimiter is `:` by default, optionally `%` instead.
607 +
608 + The `start-marker` and `end-marker` are not part of the quote.
609 + The `start-marker` and `end-marker` must be either
610 + on separate lines from the quote,
611 + or on the same line as the (single-line) quote.
612 + When the `start-marker` and `end-marker` are on separate lines from the quote,
613 + the lines containing the `start-marker` and `end-marker`
614 + are not part of the quote.
615 +
616 + Suppose `file.txt` contains the following lines:
617 +
618 +     (start)XCite can recognize this as a quote.(end)
619 +
620 +     // The marker is in this line.
621 +     XCite can recognize this as a quote.
622 +     // The marker is in this line.
623 +
624 +     // WRONG
625 +     XCite does not recognize this as a quote. // WRONG
626 +
627 + * The citation string `[file.txt:start:end]` resolves to
628 +   `XCite can recognize this as a quote.`
629 + * The citation string `[file.txt%marker]` resolves to
630 +   `XCite can recognize this as a quote.`
631 + * The citation string `[file.txt:WRONG]` resolves to an empty string.
632 +
633 + Although XCite does resolve citations recursively, it does not check for loops.
634 + A loop occurs when a quote includes a citation quoting the original citation.
635 +
636 + If you create a citation loop,
637 + XCite processes it recursively until it runs out of memory.

657 + ## Developing UML Diagrams
658 +
659 + For UML diagrams, use PlantUML text files instead of images.
660 +
661 + [PlantUML](http://plantuml.sourceforge.net/)
662 + is an open source tool written in Java to create UML diagrams from text files.
663 +

665 + when constructing sequence diagrams.

667 + The text files are rendered as PNG images where they are found,
668 + so put your PlantUML `.txt` files in the `images/` directory for your book.
669 + Then reference the `.png` version as if it existed already.

671 +     <mediaobject xml:id="figure-my-sequence-diagram">
672 +      <alt>Generated sequence diagram</alt>

674 +       <imagedata fileref="images/my-sequence-diagram.png" format="PNG" />

677 +       <para>
678 +        The sequence diagram is described in images/my-sequence-diagram.txt.
679 +       </para>
680 +      </textobject>
681 +     </mediaobject>

683 + Your PlantUML text files must have extension `.txt`.
