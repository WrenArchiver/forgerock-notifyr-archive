1 + # Specification of the CREST HTTP protocol
2 +
3 + CREST defines 6 core operations, Create, Read, Update, Delete, Patch, and Query, as well
4 + as an extensible operation called Action. These are referred to collectively as CRUDPAQ.
5 + Each operation comprises of a request and one or more responses. Each request defines
6 + the set of paramepters that may be used with it. Of most interest to developers (you!)
7 + is how CREST maps to HTTP. In particular, how the CREST operations map to HTTP methods,
8 + headers, query parameters, content, and status codes.
9 +
10 + ## Create (server provided resource ID)
11 +
12 + A POST create request should be used when the resource to be created will have a server
13 + provided resource ID (e.g. a UUID). It may also be used in cases where the resource ID
14 + is client provided, but only as long as the resource ID is included as a field within the
15 + new resource.
16 +
17 + POST create requests should target the resource container and must include the parameter
18 + `_action=create`.

69 + ## Create (client provided resource ID)
70 +
71 + A PUT create request should be used when the resource to be
72 + created has a known resource ID. The resource ID is
73 + specified as part of the target resource name, and may also
74 + be included as a field within the new resource, in which
75 + case the two must be equivalent.
76 +
77 + PUT create requests should target the new resource and
78 + include the HTTP header `If-None-Match: *`.

1 + # ForgeRock Commons REST (CREST)
2 +
3 + ## What is CREST?
4 +
5 + CREST stands for "Commons REST". CREST is the collective name given to 1) a
6 + RESTful [protocol](Protocol.md) specified by ForgeRock, 2) a [Java framework](json-resource/README.md) for implementing
7 + CREST applications, and 3) a CHF based [HTTP implementation](json-resource-http/README.md) of the RESTful protocol.
8 +
9 + CREST is not a generic framework for implementing any REST API. It is not another
10 + Restlet or JAX-RS. It is also not a complete REST API because it only defines the protocol
11 + for interacting with resources. Specifically, it only defines the operations (verbs) and their parameters, and
12 + does not specify any endpoints nor specific types of resources (users, groups, etc). It is the responsibility of CREST
13 + applications to specify the following:
14 +
15 + * the end-points - top-level end-points may be defined using a
16 +   [Router](json-resource/src/main/java/org/forgerock/json/resource/Router.java)
17 + * the schema used for resources passed to and from the CREST endpoints. At some
18 +   point we do expect to define a common schema for common types of resource, such as
19 +   users and groups
20 + * the set of extended "actions" which may be performed against a resource. Only
21 +   the core operations of "create", "delete", "patch", "query", "read", and "update"
22 +   are specified.
23 +
24 + The CREST project is split into several modules, of which the following three are most important:
25 +
26 + * [json-resource](json-resource/README.md) - Java based framework for implementing CREST applications
27 + * [json-resource-http](json-resource-http/README.md) - HTTP based client / server implementation of CREST using CHF
28 + * [json-resource-examples](json-resource-examples/README.md) - runnable examples illustrating CREST application
29 + development

1 + # ForgeRock Commons REST (CREST) application framework
2 +
3 + ## What is the CREST application framework?
4 +
5 + For an overview of CREST and its related HTTP protocol please refer to the [README](../README.md) in the parent project.
6 +
7 + This module provides a Java framework for implementing CREST applications. Generally speaking a CREST application
8 + comprises of:
9 +
10 + * (RequestHandlers)[src/main/java/org/forgerock/json/resource/RequestHandler.java]
11 +  - are responsible for processing a Request and yielding a Response. Examples of handlers include Routers,
12 +  (databases)[src/main/java/org/forgerock/json/resource/MemoryBackend.java], and proxies
13 + * (Routers)[src/main/java/org/forgerock/json/resource/Router.java] - are a specific type of handler which are
14 + responsible for directing requests to subordinate handlers based on criteria such as the request's URL
15 + (ResourcePath) or the Request's required API version
16 + * (Filter chains)[src/main/java/org/forgerock/json/resource/FilterChain.java] - are another common type of handler
17 + comprising of a series of (filters)[src/main/java/org/forgerock/json/resource/Filter.java] and a terminal handler. A
18 + filter chain represents a chain of processing, where each filter may perform a role such as logging, access control,
19 + transformations, etc.
20 +
21 + Take a look at the [example applications](../json-resource-examples/README.md) for further help.
22 +
23 + ## Getting started
